# План исправлений по замечаниям (выполнено)

## 1. Смета: два сообщения Greenwich + текст вложения
- **Проблема:** При первой отправке сметы Greenwich получает только файл, без информационного сообщения «проверьте смету». При повторной отправке (после правок) сообщение есть.
- **Причина:** В `sendToGreenwichUsers` сначала отправляется текст, затем файл; возможен race или сбой первой отправки.
- **Действия:**
  - Гарантировать порядок: сначала отправлять текстовое сообщение и дожидаться его для каждого пользователя, затем отправлять файл.
  - Заменить подпись файла: не «Смета по заявке {orderId}», а человекопонятный текст, например «Смета: {eventName или customer.name}» или просто «Файл сметы».

## 2. Текст сообщения с файлом сметы
- Заменить `Смета по заявке ${order.id}` на что-то вроде «Смета: {название мероприятия / клиент}» или «Файл сметы» (использовать `order.eventName` / `order.customer.name`).

## 3. Сообщение после приёмки: статусы позиций
- **Проблема:** В сообщении сотруднику Greenwich после приёмки нет сводки по статусам (в норме / сломано / потеряно и т.д.).
- **Действия:**
  - В API check-in при вызове `notifyCheckInClosed` передавать заказ с `returnSplits` (phase CHECKED_IN).
  - Расширить тип `OrderForNotify` (или передать отдельно) данными по returnSplits.
  - В `notifyCheckInClosed` формировать блок: по каждой позиции/линии — сколько OK, NEEDS_REPAIR, BROKEN, MISSING, и добавлять в текст сообщения.

## 4. Подборки = категории (каталог и CRUD)
- **Проблема:** В CRUD подборок список пустой; в каталоге уже есть «категории» (Фотозоны, Игры, Мебель…). Подборка = категория.
- **Текущее состояние:** Каталог использует модель Category (таблица Category); в сиде заполняются Category и ItemCategory. Модель Collection (подборки) в сиде не заполняется.
- **Действия:**
  - В сиде после создания Category и привязки ItemCategory создать записи Collection с теми же названиями (Фотозоны, Игры, Мебель, Декор, Свет, Текстиль).
  - Для каждого Item, привязанного к Category, создать связь CollectionItem с соответствующей Collection (по совпадению имени категории и подборки).
  - В результате в CRUD подборок появятся те же подборки, что и в каталоге; данные не теряются, каталог продолжает работать по Category.

## 5. Редактирование пакетов: не плоский список всего
- **Проблема:** Сейчас показывается полный список всего реквизита с полем количества напротив каждой позиции; при 500 позициях это неудобно.
- **Действия:**
  - Добавить поиск по названию (фильтр списка).
  - Реализовать режим «добавить позицию»: показывать список (с поиском), при выборе позиции добавлять её в состав пакета с полем количества; внизу — список уже добавленных в пакет с возможностью изменить количество или удалить.
  - Либо оставить текущий список, но добавить фильтр/поиск и группировку по категориям для удобства.

## 6. Редактирование позиции: чекбоксы вместо multi-select
- **Проблема:** Выбор подборок и категорий через multi-select (Ctrl/Shift) неудобен.
- **Действия:**
  - В форме редактирования позиции (инвентарь · позиция) заменить оба `<select multiple>` на списки с чекбоксами: для каждой категории/подборки — строка с чекбоксом и названием.

## 7. Фото в каталоге
- **Проблема:** Загруженное фото позиции отображается в редактировании, но в каталоге по-прежнему «без фото».
- **Причина:** В каталоге всегда рендерится заглушка; URL фото не используется.
- **Действия:**
  - В каталоге (карточки позиций и модалка): если у позиции есть `photo1Key`, показывать `<img src={/api/inventory/positions/${id}/photo} />`, иначе — заглушку «без фото».
  - В ItemModal тоже показывать фото по тому же URL при наличии photo1Key.
