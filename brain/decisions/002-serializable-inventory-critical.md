# ADR 002: Serializable-транзакции для резерва и конкурирующих заявок

## Статус

Принято. Явно в коде (поиск `TransactionIsolationLevel.Serializable` / `P2034`):

| Файл | Назначение |
|------|------------|
| `src/app/api/orders/route.ts` | Создание заявки |
| `src/app/api/orders/[id]/greenwich-edit/route.ts` | Редактирование со стороны Greenwich |
| `src/app/api/orders/[id]/warehouse-edit/route.ts` | Редактирование со стороны склада |
| `src/app/api/orders/[id]/quick-supplement/greenwich/route.ts` | Быстрая доп.-выдача (Greenwich) |
| `src/app/api/orders/[id]/quick-supplement/warehouse/route.ts` | Быстрая доп.-выдача (склад) |
| `src/app/api/projects/[id]/draft-order/materialize/route.ts` | Materialize demo-черновика проекта в реальные заявки |

Во всех перечисленных при **`P2034`** клиенту отдаётся **409** (повтор запроса), не сырой 500.

## Контекст

Несколько пользователей могут одновременно оформлять заявки на пересекающиеся даты и одни и те же позиции. Чтение «доступного» количества вне строгой изоляции даёт **overbooking**.

## Решение

Использовать **`prisma.$transaction(..., { isolationLevel: "Serializable" })`** на критичных путях, где считается резерв и создаются/меняются строки.

При конфликте сериализации Prisma возвращает **`P2034`** → клиенту отдавать **409** / просьбу повторить, а не 500.

## Последствия

- Небольшое увеличение нагрузки на БД при пиках; допустимо для текущего масштаба.
- Убирать изоляцию «для скорости» нельзя без модели компенсации (очередь, блокировки строк `Item`).

## Нарушать?

Только осознанно, с нагрузочным тестом и ADR.
