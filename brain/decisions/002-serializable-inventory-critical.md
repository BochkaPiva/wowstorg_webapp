# ADR 002: Serializable-транзакции для резерва и конкурирующих заявок

## Статус

Принято (см. `POST /api/orders`, правки строк заявок, quick supplement — по коду).

## Контекст

Несколько пользователей могут одновременно оформлять заявки на пересекающиеся даты и одни и те же позиции. Чтение «доступного» количества вне строгой изоляции даёт **overbooking**.

## Решение

Использовать **`prisma.$transaction(..., { isolationLevel: "Serializable" })`** на критичных путях, где считается резерв и создаются/меняются строки.

При конфликте сериализации Prisma возвращает **`P2034`** → клиенту отдавать **409** / просьбу повторить, а не 500.

## Последствия

- Небольшое увеличение нагрузки на БД при пиках; допустимо для текущего масштаба.
- Убирать изоляцию «для скорости» нельзя без модели компенсации (очередь, блокировки строк `Item`).

## Нарушать?

Только осознанно, с нагрузочным тестом и ADR.
