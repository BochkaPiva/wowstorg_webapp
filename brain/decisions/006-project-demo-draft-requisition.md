# ADR 006: Demo-заявка проекта как отдельный local draft

## Статус

Принято.

## Контекст

Пользователю нужна возможность собирать реквизит прямо внутри карточки проекта ещё до подтверждения дат.

Текущая модель `Order` для этого не подходит:

- API и БД требуют даты при создании заявки.
- Резерв и availability считаются только на конкретный диапазон.
- Блоки реквизита в смете уже завязаны на реальные `Order` и `OrderLine`.

Попытка сделать `Order` сущностью "иногда без дат" ломает инварианты склада, резерва и жизненного цикла заявки.

## Решение

Вводим отдельный project-local слой:

- `ProjectDraftOrder` — один активный demo-черновик на проект.
- `ProjectDraftOrderLine` — строки demo-черновика с `qty`, `comment`, optional `periodGroup` и снапшотом цены.

Правила:

1. Demo-черновик не попадает в `/orders`, `/warehouse/queue`, архив заявок и обычные складские статусы.
2. Demo-черновик не резервирует остатки.
3. Для выбора позиций используется тот же каталог, но без обязательных дат: UI ориентируется на `availableNow`.
4. Подтверждение дат идёт отдельным flow `materializeDraftToOrders`.
5. Materialize создаёт 1..N реальных `Order` через тот же серверный helper, что и обычный `POST /api/orders`.
6. Materialize выполняется в `Serializable` транзакции и возвращает частичный результат: что стало реальными заявками, а что осталось в draft из-за дефицита.
7. В смете demo-блок показывается отдельным типом read model (`DRAFT_REQUISITE`), чтобы не маскироваться под живую складскую заявку.

## Последствия

- Сохраняется инвариант `1 order = 1 date range`.
- Demo-flow можно развивать независимо от очереди склада.
- UX проекта становится двухслойным:
  - `REQUISITE` = реальная заявка склада.
  - `DRAFT_REQUISITE` = проектный черновик без дат.
- Нужны отдельные migration, API и activity log для demo-черновика.

## Нарушать?

Не превращать `Order` в сущность без дат без нового ADR и отдельного анализа влияния на резерв, выдачу и приёмку.
