## 3. Границы и допущения (инженерные)

1. **Доступ:** только роль **WOWSTORG** на новых маршрутах и серверных операциях. Greenwich и публичные клиенты **не** получают списки/детали проектов через новые API.
2. **Заявки без проекта:** все текущие сценарии `Order` с `projectId = null` **обязаны** вести себя как сейчас (резервы, сметы, статусы, рейтинг, инциденты).
3. **Изменения схемы `Order`:** только **аддитивные** в первой волне (`projectId` nullable + индекс). Не переписывать существующие поля и не менять семантику `OrderSource` / `OrderStatus` ради проектов.
4. **Пользователи:** 2 человека — **без** real-time collaborative editing в v1; достаточно **last-write-wins** на уровне полей + **журнал активности** для разбора «кто что менял».
5. **Безопасность на уровне приложения:** RLS в Postgres под приложение не опираемся; все проверки — **серверные** (`requireRole`, проверка владения/доступа к сущности).

## 4. Карта зависимостей

### 4.1. База данных (существующее)

| Объект | Зависимость для проектов |
|--------|---------------------------|
| **`Customer`** | Карточка проекта привязана к заказчику; те же клиенты, что и у заявок. Удаление/деактивация клиента — продумать политику (см. §8). |
| **`User`** | Ответственный за проект, автор записей журнала, загрузчик файлов. `onDelete` для проектных сущностей — **Restrict** или запрет удаления пользователя с активными проектами. |
| **`Order`** | Связь **1 проект → N заявок** через `projectId` (nullable). Логика резерва, выдачи, возврата **не** переносится в модуль проектов — остаётся в текущих сервисах заявок. |
| **`Item` / резервы** | Каталог «под проект» создаёт заявки тем же механизмом, что внешний поток; **Serializable** и `P2034` остаются обязательными. |
| **`ReminderSent`** | Привязка к `orderId`; напоминания по **проекту** в v1 не вводим без отдельного ТЗ (избегаем дублирования и путаницы). |

### 4.2. Код и модули

| Зона | Риск касания |
|------|----------------|
| Создание заявок (внешний / внутренний каталог) | Новый флоу «каталог под проект» должен **переиспользовать** существующие функции создания `Order`, а не дублировать бизнес-правила цен и резерва. |
| Quick supplement / доп. заявки | Быстрые доп.-заявки Greenwich больше не должны быть «урезанным» типом заказа по доп. услугам: разрешены те же флаги услуг, а склад обязан проставить цены до перехода в сборку. |
| Уведомления (Telegram и т.д.) | Ссылки и тексты завязаны на заявки; проекты в v1 **не** подменяют каналы уведомлений без явного решения. |
| Storage (`file-storage.ts`) | Новый bucket или согласованный префикс; **не** смешивать ключи с `item-photos` / `estimates` без префикса `projects/…`. |
| Аудит инвентаря / отчёты | Проекты не должны ломать агрегации по `Order`; при отчётах `projectId` — опциональная группировка. |

## 5. Принципы «надёжного фундамента»

1. **Схема раньше UI-полноты:** миграции + минимальные API + smoke-тест руками; затем насыщение UI.
2. **Источник правды по заявкам:** статус и резерв позиции живут в **`Order`** и **`OrderLine`**. Карточка проекта **создаёт** заявки и для блока **«Реквизит»** в смете **пишет в `OrderLine`** (очередь и склад видят то же); остальные разделы сметы — только данные проекта (см. [`projects.md`](./projects.md) § Смета).
3. **Идемпотентность там, где возможно:** повторный клик «создать черновик» не должен плодить дубликаты без явного намерения (клиентские ключи или проверка «уже есть черновик» — по мере необходимости).
4. **Журнал активности проекта (`ProjectActivityLog`):** записи **только добавление** (append-only на уровне продукта); правки/удаление — исключение и только под супер-админскую политику (в v1 можно не делать).
5. **Удаление проекта:** предпочтительно **мягкое** (`archivedAt` / `isArchived`), а не `DELETE`, пока есть связанные заявки или файлы в Storage — иначе каскады и «висячие» ключи.
6. **Файлы:** при удалении записи `ProjectFile` — **best-effort** удаление объекта в Storage (как в текущем коде фото); ошибки Storage **не** откатывают успешное удаление метаданных, если продукт так решит — зафиксировать единообразно.

## 6. Реестр рисков

| ID | Риск | Вероят. | Влияние | Митигация |
|----|------|---------|---------|-----------|
| R1 | Регрессия заявок без `projectId` | Средняя | Критич. | Автотесты/ручной регресс критических путей заказа; `projectId` только optional; не менять условия в запросах без `projectId`. |
| R2 | Дублирование логики цен/резерва в «каталоге проекта» | Средняя | Высок. | Вызов существующих серверных путей создания заказа; код review на «копипасту». |
| R3 | Гонки при двух редактирующих одну карточку | Низкая | Средн. | `updatedAt` в UI (предупреждение «данные устарели») опционально в v1; минимум — activity log. |
| R4 | Переполнение Storage / злоупотребление upload | Средняя | Средн. | Лимиты **на сервере** (число файлов, суммарный размер по проекту) из спеки; отказ с понятной ошибкой. |
| R5 | Несовпадение preview/prod БД (забыли миграцию) | Средняя | Высок. | CI или чеклист деплоя; не мержить в master без `migrate deploy` на целевой БД. |
| R6 | Долгие транзакции при создании проекта + папок + лога | Низкая | Средн. | Одна короткая транзакция Prisma на «создание каркаса»; тяжёлые вещи (upload) вне транзакции. |
| R7 | PDF/docx генерация падает по памяти на Vercel | Средняя | Средн. | Генерация в route с лимитом размера; streaming или временные файлы по паттерну существующих отчётов; таймауты. |
| R8 | Утечка данных проекта через ошибочный API | Низкая | Критич. | Все маршруты `/api/projects/**` с `requireRole("WOWSTORG")`; не отдавать проект по `orderId` без проверки прав на заказ. |
| R9 | Сиротские файлы в Storage после сбоя | Средняя | Низк. | Периодическая уборка или журнал «pending upload» — в бэклог; минимум — логирование key при upload. |
| R10 | Синк «Реквизит» проекта → `OrderLine` при конкуренции с правками заявки со склада | Средняя | Высок. | Те же **Serializable / обработка P2034**, что на путях редактирования заказа; не обходить существующий сервис обновления строк. |
| R11 | Несколько заявок — путаница синка реквизита | Низкая | Средн. | **Продукт:** по одному **блоку реквизита на заявку**, заголовок с **датами**; каждый блок ↔ свой `orderId` ([`projects.md`](./projects.md) § Уточнения п.6). |
| R12 | Разъезд формул UI/XLSX/экспорта | Средняя | Высок. | Общий helper для totals сметы проекта; не считать комиссию/налог отдельными локальными формулами в каждом слое. |
| R13 | Quick supplement с включёнными услугами уходит в сборку без цены | Средняя | Высок. | Общая server-проверка обязательных цен для enabled услуг и блокировка `start-picking` до их заполнения. |

## 7. Слабые места (осознанные компромиссы)

1. **Один статус + мяч** не отражают параллельные треки (деньги + правки сметы) — закрывается полями вроде `openBlockers` / текстовыми полями шапки (см. `projects.md`).
2. **Нет напоминаний по проекту** в MVP — риск «забыли задачу»; митигация: статусы + журнал + ручной процесс до отдельной фичи.
3. **Нет real-time** — второй пользователь видит обновления после перезагрузки/навигации; при необходимости позже — polling или SSE только для карточки проекта.
4. **Целостность «проект ↔ Storage»** без внешнего FK — только приложение; при критичности — фоновая сверка или админ-скрипт.

## 8. Конфликты с существующим функционалом — чеклист

Перед merge в `master` убедиться:

- [ ] Создание **WOWSTORG_EXTERNAL** (и прочих источников) **без** проекта не изменилось.
- [ ] Greenwich по-прежнему **не видит** проекты и не получает лишних полей в JSON заявок.
- [ ] Резерв/выдача/возврат: те же уровни изоляции транзакций, те же ответы клиенту при `P2034`.
- [ ] Сметы и фото позиций (текущие buckets) **не** пересекаются с ключами файлов проектов.
- [ ] Cron (`reminders`, `inventory-audit`) **не** зависит от наличия `Project` в схеме до готовности.
- [ ] `npm run brain:inventory` выполнен после добавления новых `route.ts` и `process.env.*`.
- [ ] Обновление реквизита из проекта **не обходит** правила резерва/количеств и не даёт «тихих» расхождений с очередью заявок.

## 9. Модель данных: правила целостности (черновик для Prisma)

До написания миграций зафиксировать в задаче:

| Тема | Рекомендация |
|------|----------------|
| `Order.projectId` | `String?`, индекс `(projectId)`, FK `ON DELETE SET NULL` **или** запрет удаления проекта с заявками — выбрать одно; **SET NULL** проще для истории заказов. |
| `Project` → `Customer` | `onDelete: Restrict` (не удалять клиента с проектами) или мягкое архивирование клиента. |
| `Project` → `User` (ответственный) | `Restrict`; смена ответственного — через UI, не через каскадное удаление. |
| Дочерние сущности проекта | `onDelete: Cascade` от `Project` **только** для чисто проектных таблиц (папки, файлы-метаданные, смета-проект, тайминг), если продукт допускает hard delete при «полном удалении проекта»; иначе — только soft delete на `Project`. |

Уточнить в реализации: нужна ли таблица связи `ProjectOrderLink` при наличии `Order.projectId` — избыточность, если достаточно обратной связи `Order` → `Project`.

## 10. API и слой сервера

- **Префикс:** `/api/projects/...` (или вложенные ресурсы), не размазывать по существующим `/api/orders/...` без необходимости.
- **Авторизация:** каждый handler — явная проверка WOWSTORG; для операций «от имени проекта» проверять существование проекта и права (в v1 все складские пользователи равны — без тонких ACL).
- **Ошибки:** 4xx с понятным телом; 5xx — лог на сервере; **не** подмешивать стектрейсы в прод-ответы.

## 11. Порядок внедрения (фазы)

1. **Фаза A — Скелет:** Prisma: `Project`, enums статуса/мяча, `Order.projectId`, минимальные поля; миграция на staging; CRUD API + пустой/простой список и карточка в UI.
2. **Фаза B — Связь с заявками:** создание заказа из проекта (каталог-проект), список заявок на карточке (read-only к существующим страницам заказа).
3. **Фаза C — Контент:** контакты, журнал, activity log.
4. **Фаза D — Файлы:** дерево папок, Storage, лимиты.
5. **Фаза E — Смета конструктор + PDF.**
6. **Фаза F — Тайминг + docx.**

Параллельность: D и C можно частично совмещать после стабильного A.

### 11.1 Статус реализации (ведётся в коде)

| Фаза | Сделано | Следующий шаг |
|------|---------|----------------|
| **A** | `Project`, API CRUD, `/projects`, архив, `ReminderSent` в схеме | — |
| **B** | `POST /api/orders` + `projectId`, каталог/корзина `?projectId=`, список заявок на проекте, **`GET /api/orders/[id]`:** поле `project` + скрытие `projectId` для GREENWICH; проектные заявки подсвечены в очереди и архив вынесен во вкладку на `/warehouse/queue` | Дальше только локальные UX-улучшения, без смены контракта связи `Order` ↔ `Project`. |
| **C** | Журнал активности + **`ProjectContact`** / **`ProjectCommunicationEntry`**, API `/api/projects/[id]/contacts`, записи переговоров; события контактов в `ProjectActivityLog`; категории контактов `DECISION_MAKER` / `CONTRACTOR` / `VENUE` / `OTHER` и цветовое разделение в UI | Вложения к записям переговоров — позже. |
| **D** | env бакета в `.env.example`, API файлов/папок, upload/list, лимиты по спеке | Дальше только визуальные улучшения UI файлов без изменения серверной модели. |
| **E** | Таблицы сметы проекта / версии, явная **основная версия**, удаление версий, ручной импорт из выбранных заявок проекта, live read model для requisites-блоков от `Order` / `OrderLine`, XLSX-экспорт проектной сметы, локальный draft для **локальных** разделов с явным `Сохранить смету`, отдельный `ProjectDraftOrder` / `ProjectDraftOrderLine` для demo-реквизита без дат и materialize в 1..N реальных `Order`, единый CTA `Каталог → реквизит`, project-demo catalog без дат и project-dated catalog с автоподстановкой дат мероприятия | Дальше можно развивать confirm-flow и авто-раскладку по периодам; инвариант остаётся прежним: реальные заявки живут только в `Order`, demo-слой отдельно, а target version сметы передаётся явно из project-flow. |
| **F** | `ScheduleDay` / `ScheduleSlot`, экспорт docx по спеке, UX предзаполнения следующего слота и двухколоночное представление списка, Telegram cooldown-уведомления по таймингу/смете/файлам, локальный draft тайминга с явным `Сохранить тайминг` | Дальше возможны только шаблоны экспорта и полировка редактора; клиент работает по модели draft -> Save all, без reload на каждое действие. |

Инвариант: в ответах для **GREENWICH** не отдаём поля проекта (`projectId` / `project`), чтобы не расширять поверхность данных вне продукта.

## 14. Зафиксированные технические решения

| Тема | Решение |
|------|---------|
| Bucket файлов проектов | **`project-files`**, env **`SUPABASE_STORAGE_PROJECTS_BUCKET`** (отдельно от фото и смет заявок) — см. [`projects.md`](./projects.md) § Уточнения п.7. |
| Архив в БД | Поле **`Project.archivedAt`** (`DateTime?`); бизнес-статус проекта не заменяет архив. |
| Архивация проекта | Разрешать архив только если все связанные заявки в `CLOSED` / `CANCELLED`; проверка обязательна на сервере, UI лишь дублирует ограничение. |
| Связь заказ–проект | Достаточно **`Order.projectId`**; отдельная таблица связи не нужна. |
| Удаление проекта | Жёсткое удаление не в MVP; **архив** + при необходимости позже политика FK на заказы ([`projects.md`](./projects.md)). |
| Основная версия сметы | В проекте допускается одна явная primary-версия; она открывается и экспортируется по умолчанию, при удалении primary нужно переназначение на последнюю оставшуюся версию. |
| Блоки `Реквизит` в смете | Для чтения и UX это live read model от связанной `Order`; правки идут через существующие order-edit guards, а не через прямую запись в копии строк сметы. |
| Draft workflow сметы и тайминга | Добавление/редактирование в UI сначала живёт в локальном draft (`localStorage` в браузере); запись в БД делается отдельным batch `PATCH` на `/api/projects/[id]/estimate` и `/api/projects/[id]/schedule`. |
| Demo-реквизит проекта без дат | До подтверждения дат реквизит живёт в `ProjectDraftOrder`; он не резервирует остатки и не попадает в очередь склада. Вход в него идёт через общий CTA `Каталог → реквизит`, а выбор `demo` / `real` зависит от подтверждения дат проекта и наличия draft. У строк есть **`plannedDays`** для предварительной сметы (`qty × дней × цена/день`); materialize в реальные заявки идёт отдельным server-flow с `Serializable`, а в read model сметы demo-блок маркируется как отдельный тип. В карточке проекта demo-заявка раскрывается со **списком позиций**; удаление — **`DELETE /api/projects/[id]/draft-order`**. В UI materialize задаёт **интервалы по каждой строке**, после чего клиент группирует одинаковые интервалы в `periods[]`, сохраняя инвариант `1 order = 1 date range`. |
| Unified project requisite flow | Для проекта всегда один primary entry point. `demo`-корзина и `real`-корзина разделены по storage scope (`project-demo:<projectId>` и `project:<projectId>`), а новые реальные блоки реквизита должны писаться в явно выбранную пользователем версию сметы, а не в серверный default. |
| Project Telegram notifications | Уведомления по проекту идут best-effort через `scheduleAfterResponse`; для шумных блоков (`estimate`, `schedule`, `files`) cooldown хранится в БД (`ProjectNotificationCooldown`) на 15 минут. |
| Доп. услуги и прибыль | Клиентские цены услуг (`deliveryPrice` / `montagePrice` / `demontagePrice`) и внутренние затраты (`deliveryInternalCost` / `montageInternalCost` / `demontageInternalCost`) должны расходиться только в местах, где UI явно показывает выручку vs себестоимость; прибыль по проектной смете считается через `costInternal`, а обязательность клиентской цены проверяется на переходах заказа. |
| Пустые draft-версии сметы | Новая версия сметы не должна получать обязательный `LOCAL`-раздел по умолчанию; пользователь сам решает, создавать ли локальные блоки. |
| Save local sections | `PATCH /api/projects/[id]/estimate` остаётся batch-сохранением только для `LOCAL` / `CONTRACTOR`, но клиент и сервер обязаны защищаться от случайного удаления всех локальных секций из-за stale `localStorage`: пустой destructive draft нельзя молча применять без явного подтверждения. |

**Закрыто в спеке:** заявка из каталога проекта — только `WOWSTORG_EXTERNAL` и full price; смета проекта и смета/строки заявки — разные сущности; блоки реквизита 1:1 с заявкой ([`projects.md`](./projects.md) § Уточнения).
