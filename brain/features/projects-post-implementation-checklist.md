## 3. Чеклист: что сделать вам после мержа / перед продом

### База данных

1. На **каждой** среде (staging → production) применить миграции Prisma: `npm run db:deploy` или ваш CI-эквивалент.
2. Убедиться, что нет «висящих» частичных миграций: `npx prisma migrate status`.
3. При необходимости — сид демо-данных (проекты в сид не обязательны).

### Supabase Storage

1. Создать bucket **`project-files`** (или имя из `SUPABASE_STORAGE_PROJECTS_BUCKET`).
2. Политики доступа — в духе приватных бакетов приложения (доступ через сервер с service role), не путать с публичными URL без необходимости.
3. См. подробнее [`projects-predeploy-checklist.md`](./projects-predeploy-checklist.md).

### Переменные окружения (Vercel / сервер)

1. **`SUPABASE_STORAGE_PROJECTS_BUCKET`** — задана для Preview и Production и совпадает с реальным именем бакета в соответствующем проекте Supabase.
2. **`SUPABASE_URL`**, **`SUPABASE_SERVICE_ROLE_KEY`** — как для остальных загрузок.
3. После изменения env — **пересобрать** деплой.

### Синхронизация документации для агентов

1. Выполнить **`npm run brain:inventory`** после добавления маршрутов API или новых `process.env.*`, чтобы инвентарь в `brain/` оставался актуальным.

### Дымовые проверки (ручные, WOWSTORG)

1. Открыть **«Проекты»** → создать проект → убедиться, что появились папки **Сметы / Дизайн / Документация**.
2. Загрузить файл в папку → скачать → удалить; проверить лимиты (сообщение при превышении 15 файлов или 200 МБ).
3. **Каталог** с `?projectId=` → оформить заявку → в карточке проекта видна заявка, в журнале — «Создана заявка».
4. Отменить эту заявку (допустимые статусы) → в журнале проекта — «Заявка реквизита отменена».
5. Архивировать проект → редактирование и загрузки недоступны, просмотр и скачивание — по продуктовой политике (сейчас: файлы и журнал читаются).
6. Смета: создать версию, править строку, скачать PDF; после заявки из проекта — блок «Реквизит» с копией позиций.
7. Тайминг: день + слоты → скачать .docx, открыть в Word.

### Регрессия (не сломать старое)

1. Создать **внешнюю заявку без проекта** — как раньше (заказчик, резерв, статусы).
2. **Greenwich**: не видит проекты; в API заявки для Greenwich нет лишнего `project` (кроме согласованного контракта для склада).
3. Напоминания / аудит инвентаря / cron — без ошибок из-за новых таблиц.

### Перед merge в `master` (из foundation)

Сверка с [`projects-foundation.md`](./projects-foundation.md) §8: создание заявок без проекта, Greenwich, резервы/P2034, ключи Storage, `brain:inventory`.

## 4. Зависимости npm

После `git pull` выполните **`npm install`** — для сметы и тайминга: **`pdf-lib`**, **`@pdf-lib/fontkit`** (встраивание шрифта в PDF), **`docx`**. В репозитории лежат **`src/server/fonts/NotoSans-*.ttf`** (SIL OFL) — без них PDF с кириллицей не соберётся.
