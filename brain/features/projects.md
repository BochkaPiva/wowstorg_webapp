# Проекты / мероприятия (Wowstorg) — спецификация v1 (черновик)

Дата: 2026-04-05. Доступ: **только WOWSTORG** (склад/админ). Greenwich и внешние клиенты **не** видят модуль.

**Инженерный фундамент (риски, зависимости, конфликты с текущим кодом):** [`projects-foundation.md`](./projects-foundation.md).

## Уточнения (зафиксировано)

1. **Архив проектов** — полные данные, **развёрнутый просмотр**; в архиве **только просмотр**, без редактирования. **Технически:** поле **`archivedAt`** (`DateTime?`): `null` = в активном списке, не `null` = в архиве (фильтр для UI). **Бизнес-статус** (`LEAD` … `COMPLETED`) остаётся отдельно: архив — операция «убрать с основного экрана», не замена статуса. **Доп. правило v1:** проект нельзя архивировать, пока у него есть связанные заявки не в финальных статусах `CLOSED` / `CANCELLED`.

2. **Заказчик и ЛПР** — у проекта **одна компания-заказчик** (`Customer`). У неё может быть **несколько ЛПР** — это **`ProjectContact`** (ФИО, контакты, роль). У каждого ЛПР — **свой поток коммуникаций** (`ProjectCommunicationThread` + записи **`ProjectCommunicationEntry`**): по сути **ручные карточки** «кто что написал/договорились», с автором и временем.  
   **Идеи UI (v1 можно упростить):** вкладка «Коммуникации» с **колонками или аккордеоном по ЛПР**; сверху **единая лента** всех записей с **фильтром по контакту**; кнопка «Быстрая запись» с выбором ЛПР по умолчанию (последний использованный); **закреп** важной записи вверху потока (позже); дубликат текста из последней записи как черновик (опционально). Примитивный v1: список контактов слева → справа лента карточек только для выбранного ЛПР + форма добавления.

3. **`Customer` vs сотрудники** — `Customer` = компания-контрагент; **Ваусторг** в процессе = **`User`**. Для двух человек достаточно текущей модели.

4. **Заявка из проекта (каталог-проект)** — **нет** выбора «для Greenwich»; заявка всегда как **сторонний / третьи лица**: источник **`WOWSTORG_EXTERNAL`**, **`payMultiplier` = 1** (полная цена), та же семантика, что у внешней заявки склада без коэффициента Greenwich.

5. **Смета: старт из заявки** — у **проекта** и у **заявки** **разные сущности**. При **создании** заявки из проекта позиции заказа **однократно** копируются в конструктор сметы проекта (**без автосинка** при последующих правках заказа, пока не оговорено иное). Для уже созданных версий есть отдельное ручное действие: подтянуть блоки реквизита из **выбранных** заявок проекта в текущую версию.

6. **Смета: блок «Реквизит» vs остальные разделы** — в конструкторе сметы проекта выделяется тип раздела **«Реквизит»** (аренда/позиции, уходящие в заявку). **Редактирование строк внутри такого блока** обновляет **`OrderLine` той заявки**, с которой блок связан. **Остальные разделы** (субподрядчики, техничка и т.д.) — **только проект**, в заявку не проталкиваются.  
   **Несколько заявок:** для каждой заявки — **отдельный блок реквизита** в конструкторе (1 блок = 1 заявка). В **заголовке раздела** явно указываются **даты** этой заявки (и при необходимости короткий идентификатор заявки в UI), чтобы не путать блоки.

7. **Технические решения (оптимальные по умолчанию)**  
   - **Storage:** отдельный bucket **`project-files`** (переменная **`SUPABASE_STORAGE_PROJECTS_BUCKET`**), не префикс в `item-photos` / `estimates`. Так проще квоты, политики, бэкапы и меньше риск пересечься с ключами каталога. В staging/prod создать bucket с тем же именем (как для фото/смет).  
   - **Архив:** см. п.1 — **`archivedAt`**, без смешивания с enum статуса как единственным признаком архива.

## Принятые решения (из переговоров)

| Тема | Решение |
|------|---------|
| Связь с заявками | **1 проект → N заявок** на реквизит; заявка **может быть без** проекта |
| Создание заявки из проекта | Отдельный **каталог под проект** (без поля заказчика; даты только в этом флоу); **только внешняя заявка**: `WOWSTORG_EXTERNAL`, **full price** (`payMultiplier` = 1), **без** варианта «для Greenwich» |
| Архив проектов | Полные данные; **развёрнутый просмотр**; **только чтение**; признак архива — **`archivedAt`** (отдельно от бизнес-статуса) |
| Архив проекта и заявки | Проект архивируется только если все его заявки в `CLOSED` / `CANCELLED`; архив заявок открывается как вкладка на странице очереди |
| Файлы проектов (Storage) | Отдельный bucket **`project-files`**, env **`SUPABASE_STORAGE_PROJECTS_BUCKET`** |
| Заказчик / ЛПР | **Один** `Customer`; **несколько** `ProjectContact` (ЛПР); история коммуникаций — **ручные записи** по потоку на контакт (см. § Уточнения п.2) |
| Смета: старт из заявки | Разные сущности; при **создании** заказа — **однократная** копия строк в конструктор проекта |
| Смета: блок реквизита | **Один блок реквизита на заявку**; заголовок блока — **явные даты** (и при необходимости ссылка на заявку); правки строк блока → **`OrderLine`** этой заявки |
| Смета: прочие разделы | Субы, техничка и т.д. — **только проект**, без записи в заявку |
| Реквизит под проект | **Опционально**; без дат в каталоге — не создаём заявку (резерв только при известных датах) |
| Лимит файлов | ориентир **≤15 файлов** на проект; **≤200 МБ** суммарно на проект (с запасом) |
| Смета PDF | Только из **внутреннего конструктора**; по умолчанию экспортируется **основная** версия сметы; загруженный файл сметы — **хранилище**, без парсинга в строки |
| Файлы | Единая **файловая зона** с папками; стартовый набор из **3 папок** + полное **переименование / создание / удаление** папок и файлов |
| Стартовые папки | **Сметы**, **Дизайн**, **Документация** |
| Экспорт тайминга | **.docx**: по дням, таблица колонки **Интервал** \| **Описание сценария**; вёрстка минимальная, заложить возможность шаблона позже |
| Пользователи | **2 человека** — без усложнения real-time collaboration в v1 |

## Сущности (черновик модели)

- **Project** — карточка мероприятия: название, компания-заказчик (`Customer`), статус, «мяч», даты мероприятия (опционально / уточняемые), ответственный (user), **`archivedAt`** (архив = не null), createdAt…  
  - Рекомендуемые поля для ясности (см. матрицу кейсов внизу): **`eventDateNote`** (текст), **`eventDateConfirmed`** (bool), **`openBlockers`** (textarea), **`internalSummary`** (textarea).
- **ProjectContact** — представитель: ФИО, телефон, email, роль/примечание; **`isActive`**; привязка к проекту
- **ProjectCommunicationThread** — 1 поток на контакт (или один общий, если контакт один)
- **ProjectCommunicationEntry** — записи журнала (текст, автор, время, опционально вложения позже)
- **ProjectFolder** — папка в дереве файлов проекта (parent nullable для корня; порядок)
- **ProjectFile** — файл: storage key, имя, размер, mime, папка, uploadedBy, version/replace chain при «замене»
- **EstimateVersion** — версия сметы из конструктора: номер версии, признак **основной** версии, snapshot JSON или нормализованные таблицы (разделы + строки)
- **EstimateLine** (если не JSON): раздел, порядок, №, название, описание, тип, costClient, costInternal
- **ScheduleDay** — день мероприятия (дата или порядковый «день 1»)
- **ScheduleSlot** — интервал (start-end текстом или time), описание
- **ProjectOrderLink** — связь `Order.id` ↔ `Project.id` (N заявок)
- **ProjectActivityLog** — неизменяемые записи: кто, когда, что изменил (статус, файл, смета, заявка…)

`Order`: добавить опциональное **`projectId`** (nullable); миграция только additive.

## Статусы проекта (v1) и «мяч»

Статус — **один** на проект. **Мяч** (`ball`) — отдельное поле для наглядности (можно подставлять по умолчанию при смене статуса, править вручную).

### Рекомендуемый enum статусов

| Код | Название (UI) | По умолчанию «мяч» | Смысл |
|-----|----------------|-------------------|--------|
| `LEAD` | Лид / первичный запрос | `CLIENT` | Входящий запрос, ещё не КП |
| `BRIEFING` | Сбор брифа / уточнение задачи | `CLIENT` | Ждём вводные, ТЗ от клиента |
| `INTERNAL_PREP` | Внутренняя подготовка | `WOWSTORG` | Считаем смету, ищем субов, без ответа клиенту |
| `PROPOSAL_SENT` | КП / смета отправлена | `CLIENT` | Ждём реакцию / согласование |
| `PROPOSAL_REVISION` | Правки КП / переговоры | `WOWSTORG` | Клиент попросил изменения — дорабатываем мы |
| `CONTRACT_PREP` | Подготовка договора | `WOWSTORG` | Готовим/согласовываем текст |
| `CONTRACT_SENT` | Договор направлен клиенту | `CLIENT` | Ждём подпись / замечания |
| `CONTRACT_SIGNED` | Договор подписан | `WOWSTORG` | Юридически стартовали, ведём реализацию |
| `PREPRODUCTION` | Предпродакшн | `WOWSTORG` | Субы, согласования, закупки, без дня Х |
| `AWAITING_CLIENT_INPUT` | Ждём данные от клиента | `CLIENT` | Логистика, списки, доступы — упёрлись в клиента |
| `AWAITING_VENDOR` | Ждём субподряд / площадку | `VENDOR` | Внешний исполнитель тормозит |
| `READY_TO_RUN` | Готово к проведению | `WOWSTORG` | Всё собрано, ждём дату или финальный чек |
| `LIVE` | Проведение / монтаж / день Х | `WOWSTORG` | Активная фаза события |
| `WRAP_UP` | Закрытие | `WOWSTORG` | Демонтаж, акты, финансы |
| `COMPLETED` | Завершён | `NONE` | Архив успеха |
| `ON_HOLD` | Пауза | `CLIENT` или `WOWSTORG` | Явно кто инициировал — правится вручную |
| `CANCELLED` | Отменён | `NONE` | Без продолжения |

### Enum «мяч» (`ProjectBall`)

- `CLIENT` — ждём клиента  
- `WOWSTORG` — наш ход  
- `VENDOR` — суб / подрядчик  
- `VENUE` — площадка (если отделите от VENDOR)  
- `NONE` — нет активного ожидания (завершено / отмена)

*Примечание:* `VENUE` можно в v1 не выводить в UI и маппить на `VENDOR`, если не хотите дробить — тогда убрать из enum.

### Редактирование списка статусов

В v1 — **фиксированный enum в Prisma** + человекочитаемые подписи в UI. Если позже понадобится «редактировать из админки» — отдельная таблица `ProjectStatusDefinition` и миграция; закладывать сейчас не обязательно.

## Смета (конструктор)

- Разделы (название, порядок) + строки (поля из ТЗ: №, название, описание, тип, цена клиента, себестоимость).  
- **Тип раздела:** минимум два класса — **«Реквизит»** (привязан к **одной** `Order` через `orderId`; синхрон строк с `OrderLine`, см. § Уточнения п.6) и **«Локальный»** (субподрядчики, техничка и пр.). При **N** заявках — **N отдельных блоков/разделов реквизита**, в каждом заголовке **явно даты** (и при необходимости обозначение заявки); **1:1 блок ↔ заявка**.  
- **Версии:** одна версия может быть явно назначена **основной**; именно она открывается по умолчанию и считается дефолтной для PDF, если версия не выбрана явно. Новая версия может быть пустой или дубликатом текущей. Разрешены действия: назначить основной, удалить версию (кроме последней) и вручную импортировать блоки реквизита из выбранных заявок проекта.  
- **PDF клиенту:** только из конструктора; показывать до **цены клиента**; блок **комиссия агентства 15%** от суммы услуг (без комиссии) + строка комиссии + **итог с комиссией**. (Точку отсчёта «сумма услуг» зафиксировать в коде: сумма строк «цена клиента».)  
- Загруженная смета — файл в папке **Сметы**; действия: скачать, заменить, удалить.

### UI панели сметы (`ProjectEstimatePanel`)

Реализация: **`src/app/projects/[id]/ProjectEstimatePanel.tsx`** (вкладка «Смета» в рабочей зоне карточки проекта).

- **Заголовок «Смета проекта»:** пояснение про живые блоки реквизита, локальные разделы как черновик с batch-сохранением, комиссию 15% и попадание её в **XLSX** — не длинным абзацем в потоке, а в **легенде по кнопке «?»** (открывается при наведении/focus), по тому же паттерну, что легенда у блока «Заявки реквизита» на странице проекта.
- **Версия сметы и «Действия»:** выпадающие списки; глобальный обработчик закрытия должен срабатывать на **`mousedown` вне контейнера** (с `ref` + `contains`), а не на `click` по `window` в том же цикле событий — иначе меню сразу закрывается при открытии.
- **Секция типа «Реквизит»:** свёрнутая шапка (`<details>`) уже показывает бейджи, **название/номер заявки**, площадку и **диапазон дат**; отдельные плашки «Связанная заявка» и «Период» в теле секции **не дублируются**. Справа в шапке: **«Открыть заявку»**, затем **«Сохранить заявку»** (если заявка редактируема из сметы). Отдельная кнопка «Полная заявка» не нужна — дублирует «Открыть заявку». **Индикатор** «можно/нельзя редактировать из сметы» (зелёный/красный круг) — **в шапке секции** рядом с заголовком, с той же легендой по hover. Ограничения количества по складу остаются в логике поля (`min`/`max`), **без** второй строки-подписи под «Кол-во».

## Тайминг-сценарий

- Дни → слоты: **интервал** (строка, напр. `09:00–10:30`) + **описание**. В UI слоты показываются как двухколоночный список `интервал | описание`, а новый слот предзаполняется от окончания предыдущего с коротким шагом по времени.  
- Экспорт **.docx**: для каждого дня заголовок даты, затем таблица двух колонок. Библиотека на выбор при реализации: `docx` / `docxtemplater` / генерация через шаблон — зафиксировать в задаче на разработку.

## Хранилище

- Отдельный bucket **`project-files`** (`SUPABASE_STORAGE_PROJECTS_BUCKET`); лимит **200 МБ на проект** проверять на сервере при upload. Настройки bucket (приватный, лимиты MIME/размера) — по аналогии с `item-photos` / `estimates`.

## MVP vs позже

**MVP (рекомендация):**

1. CRUD проекта + статус + мяч + лог действий  
2. Контакты + журнал по контакту (текстовые записи)  
3. Дерево папок/файлов со стартовыми тремя папками  
4. Конструктор сметы v1 + PDF + версии  
5. Привязка N заявок; кнопка «реквизит» → каталог-проект → создание `Order` с `projectId`  
6. Тайминг + экспорт docx (простой)

**Позже:** шаблон оформления Word/PDF, импорт xlsx в строки сметы, напоминания, аналитика по проектам.

## Матрица кейсов: хватает ли карточки на «всё ясно»

Ниже — типовые ситуации ивент-агентства и то, **чем текущая структура их закрывает**. Где есть пробел — **рекомендация по усилению** (лучше заложить в v1 как поля шапки или один блок «Сводка»).

### A. Типы мероприятий и сценарии работы

| Кейс | Статус / мяч | Смета | Файлы | Тайминг | Коммуникации | Заявки реквизита | Замечание |

| Промо в ТЦ, много субов | `AWAITING_VENDOR` / мяч VENDOR | ✅ блоки тех/орг | ✅ акты субов в Документация | ✅ монтаж/слоты | ✅ «ждём площадку» в журнале | ✅ отдельная заявка под выезд | Ок. |

| Клиент пропал / не отвечает | `ON_HOLD` + мяч CLIENT | — | — | — | последняя запись «нет ответа» | — | **Напоминание** — в «Позже»; в v1 достаточно журнала + статус. |
| Несколько юрлиц (мы + суб счета) | `CONTRACT_SIGNED` | ✅ себестоимость / клиент | ✅ папки по типам | — | ✅ | — | Если нужно **раздельно хранить «наш договор» и «суб»** — подпапки в Документация или теги имён файлов; модель не мешает. |
| Параллельно: ждём деньги И правки сметы | один статус | ⚠️ | ✅ | — | ✅ | — | **Ограничение:** один статус не отражает два трека. Решение: в шапке **два чек-флага/подсказки** «Оплата: ждём» / «Документы: ждём» или свободное поле «Открытые блокеры» (см. § «Усиления»). |

### B. Документы и сметы

| Кейс | Покрытие | Пробел / усиление |
|------|----------|-------------------|
| КП v1, v2, v3 | `EstimateVersion` + PDF из конструктора | В карточке показывать **текущую версию + список версий** одной строкой. |
| КП уехало клиенту файлом из Excel | Файл в **Сметы** | Нельзя сравнить с конструктором автоматически — ок; в UI подпись «только файл». |
| НДС / без НДС в договоре | Документ + запись в журнале | При необходимости — поле **«Условия оплаты / НДС»** текстом в шапке. |
| Частичная предоплата / этапы | Журнал + файлы счетов | Для наглядности v1: поле **«Финансы (свободный текст)»** или чеклист в шапке. |

| Один контакт | Один `ProjectCommunicationThread`. |

| Кейс | Покрытие |
|------|----------|
| Несколько выездов (монтаж + день Х) | Несколько `Order` с `projectId` — в карточке список с датами и статусами заказов (как сейчас в системе). |

### E. Где структуры **мало** для «максимальной информативности»

Один **статус** + **мяч** не передают одновременно «и деньги ждём, и клиент молчит». Это не баг спеки — это сознательное упрощение.

**Рекомендуемые усиления карточки (v1, лёгкие поля):**

1. **`eventDateNote` + `eventDateConfirmed` (bool)** — черновая дата или диапазон текстом + флаг «дата подтверждена»; смены писать в `ProjectActivityLog`.  
2. **`openBlockers` (textarea)** — одна строка «что мешает прямо сейчас» (видно в шапке без чтения всего журнала).  
3. **`internalSummary` (textarea)** — «суть в двух предложениях» для коллеги, кто открыл карточку первый раз.  
4. **Контакт `isActive`** — не путать старых ЛПР с новыми.  
5. **Блок «Заявки»** в UI: таблица `id`, даты, статус заказа, ссылка на существующую карточку заказа — чтобы не дублировать логику склада. В очереди склада проектные заявки дополнительно выделяются фиолетовым акцентом и названием проекта.

**Позже (не блокер ясности v1):** деньги в цифрах (счёт/оплачено), напоминания, теги проектов, связь «родительский / дочерний» проект для этапов.

### Вывод

Для **2 пользователей** и описанного объёма текущей структуры **достаточно**, чтобы по карточке понимать стадию, переговоры, документы, сметы, сценарий и реквизит — при условии **собранного UI**: шапка (статус + мяч + клиент + ответственный + даты/блокеры/сводка) + вкладки/секции (коммуникации, файлы, сметы, тайминг, заявки, лог).  

Без полей **§ «Усиления»** часть кейсов потребует листать длинный журнал — не ошибка модели, а снижение скорости ориентации.

## Git и окружение

- Ветка разработки модуля: **`feature/projects`**. В **`master`** мержим только когда готово к продакшену.
- **Vercel Production** (`master`): текущая Supabase и существующие переменные.
- **Vercel Preview** (пуши в `feature/projects` / PR): после создания **отдельного проекта Supabase** для задач модуля — в настройках Vercel задать для **Preview** (и при необходимости **Development**) свои `DATABASE_URL` и `DIRECT_URL` из новой БД; продакшен-БД не трогать. При использовании Storage — отдельные ключи/бакеты под staging при необходимости.
