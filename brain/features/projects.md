# Проекты / мероприятия (Wowstorg) — спецификация v1 (черновик)

Дата: 2026-04-05. Доступ: **только WOWSTORG** (склад/админ). Greenwich и внешние клиенты **не** видят модуль.

## Принятые решения (из переговоров)

| Тема | Решение |
|------|---------|
| Связь с заявками | **1 проект → N заявок** на реквизит; заявка **может быть без** проекта |
| Создание заявки из проекта | Отдельный **каталог под проект** (без поля заказчика; даты только в этом флоу); доп. услуги как у **WOWSTORG_EXTERNAL** |
| Реквизит под проект | **Опционально**; без дат в каталоге — не создаём заявку (резерв только при известных датах) |
| Лимит файлов | ориентир **≤15 файлов** на проект; **≤200 МБ** суммарно на проект (с запасом) |
| Смета PDF | Только из **внутреннего конструктора**; загруженный файл сметы — **хранилище**, без парсинга в строки |
| Файлы | Единая **файловая зона** с папками; стартовый набор из **3 папок** + полное **переименование / создание / удаление** папок и файлов |
| Стартовые папки | **Сметы**, **Дизайн**, **Документация** |
| Экспорт тайминга | **.docx**: по дням, таблица колонки **Интервал** \| **Описание сценария**; вёрстка минимальная, заложить возможность шаблона позже |
| Пользователи | **2 человека** — без усложнения real-time collaboration в v1 |

## Сущности (черновик модели)

- **Project** — карточка мероприятия: название, компания-заказчик (`Customer`), статус, «мяч», даты мероприятия (опционально / уточняемые), ответственный (user), createdAt…  
  - Рекомендуемые поля для ясности (см. матрицу кейсов внизу): **`eventDateNote`** (текст), **`eventDateConfirmed`** (bool), **`openBlockers`** (textarea), **`internalSummary`** (textarea).
- **ProjectContact** — представитель: ФИО, телефон, email, роль/примечание; **`isActive`**; привязка к проекту
- **ProjectCommunicationThread** — 1 поток на контакт (или один общий, если контакт один)
- **ProjectCommunicationEntry** — записи журнала (текст, автор, время, опционально вложения позже)
- **ProjectFolder** — папка в дереве файлов проекта (parent nullable для корня; порядок)
- **ProjectFile** — файл: storage key, имя, размер, mime, папка, uploadedBy, version/replace chain при «замене»
- **EstimateVersion** — версия сметы из конструктора: номер версии, snapshot JSON или нормализованные таблицы (разделы + строки)
- **EstimateLine** (если не JSON): раздел, порядок, №, название, описание, тип, costClient, costInternal
- **ScheduleDay** — день мероприятия (дата или порядковый «день 1»)
- **ScheduleSlot** — интервал (start-end текстом или time), описание
- **ProjectOrderLink** — связь `Order.id` ↔ `Project.id` (N заявок)
- **ProjectActivityLog** — неизменяемые записи: кто, когда, что изменил (статус, файл, смета, заявка…)

`Order`: добавить опциональное **`projectId`** (nullable); миграция только additive.

## Статусы проекта (v1) и «мяч»

Статус — **один** на проект. **Мяч** (`ball`) — отдельное поле для наглядности (можно подставлять по умолчанию при смене статуса, править вручную).

### Рекомендуемый enum статусов

| Код | Название (UI) | По умолчанию «мяч» | Смысл |
|-----|----------------|-------------------|--------|
| `LEAD` | Лид / первичный запрос | `CLIENT` | Входящий запрос, ещё не КП |
| `BRIEFING` | Сбор брифа / уточнение задачи | `CLIENT` | Ждём вводные, ТЗ от клиента |
| `INTERNAL_PREP` | Внутренняя подготовка | `WOWSTORG` | Считаем смету, ищем субов, без ответа клиенту |
| `PROPOSAL_SENT` | КП / смета отправлена | `CLIENT` | Ждём реакцию / согласование |
| `PROPOSAL_REVISION` | Правки КП / переговоры | `WOWSTORG` | Клиент попросил изменения — дорабатываем мы |
| `CONTRACT_PREP` | Подготовка договора | `WOWSTORG` | Готовим/согласовываем текст |
| `CONTRACT_SENT` | Договор направлен клиенту | `CLIENT` | Ждём подпись / замечания |
| `CONTRACT_SIGNED` | Договор подписан | `WOWSTORG` | Юридически стартовали, ведём реализацию |
| `PREPRODUCTION` | Предпродакшн | `WOWSTORG` | Субы, согласования, закупки, без дня Х |
| `AWAITING_CLIENT_INPUT` | Ждём данные от клиента | `CLIENT` | Логистика, списки, доступы — упёрлись в клиента |
| `AWAITING_VENDOR` | Ждём субподряд / площадку | `VENDOR` | Внешний исполнитель тормозит |
| `READY_TO_RUN` | Готово к проведению | `WOWSTORG` | Всё собрано, ждём дату или финальный чек |
| `LIVE` | Проведение / монтаж / день Х | `WOWSTORG` | Активная фаза события |
| `WRAP_UP` | Закрытие | `WOWSTORG` | Демонтаж, акты, финансы |
| `COMPLETED` | Завершён | `NONE` | Архив успеха |
| `ON_HOLD` | Пауза | `CLIENT` или `WOWSTORG` | Явно кто инициировал — правится вручную |
| `CANCELLED` | Отменён | `NONE` | Без продолжения |

### Enum «мяч» (`ProjectBall`)

- `CLIENT` — ждём клиента  
- `WOWSTORG` — наш ход  
- `VENDOR` — суб / подрядчик  
- `VENUE` — площадка (если отделите от VENDOR)  
- `NONE` — нет активного ожидания (завершено / отмена)

*Примечание:* `VENUE` можно в v1 не выводить в UI и маппить на `VENDOR`, если не хотите дробить — тогда убрать из enum.

### Редактирование списка статусов

В v1 — **фиксированный enum в Prisma** + человекочитаемые подписи в UI. Если позже понадобится «редактировать из админки» — отдельная таблица `ProjectStatusDefinition` и миграция; закладывать сейчас не обязательно.

## Смета (конструктор)

- Разделы (название, порядок) + строки (поля из ТЗ: №, название, описание, тип, цена клиента, себестоимость).  
- **Версии:** при «новая версия» копируем структуру или freeze snapshot предыдущей.  
- **PDF клиенту:** только из конструктора; показывать до **цены клиента**; блок **комиссия агентства 15%** от суммы услуг (без комиссии) + строка комиссии + **итог с комиссией**. (Точку отсчёта «сумма услуг» зафиксировать в коде: сумма строк «цена клиента».)  
- Загруженная смета — файл в папке **Сметы**; действия: скачать, заменить, удалить.

## Тайминг-сценарий

- Дни → слоты: **интервал** (строка, напр. `09:00–10:30`) + **описание**.  
- Экспорт **.docx**: для каждого дня заголовок даты, затем таблица двух колонок. Библиотека на выбор при реализации: `docx` / `docxtemplater` / генерация через шаблон — зафиксировать в задаче на разработку.

## Хранилище

- Новый bucket Supabase (например `project-files`) или префикс в существующем — решение при внедрении; лимит **200 МБ на проект** проверять на сервере при upload.

## MVP vs позже

**MVP (рекомендация):**

1. CRUD проекта + статус + мяч + лог действий  
2. Контакты + журнал по контакту (текстовые записи)  
3. Дерево папок/файлов со стартовыми тремя папками  
4. Конструктор сметы v1 + PDF + версии  
5. Привязка N заявок; кнопка «реквизит» → каталог-проект → создание `Order` с `projectId`  
6. Тайминг + экспорт docx (простой)

**Позже:** шаблон оформления Word/PDF, импорт xlsx в строки сметы, напоминания, аналитика по проектам.

## Матрица кейсов: хватает ли карточки на «всё ясно»

Ниже — типовые ситуации ивент-агентства и то, **чем текущая структура их закрывает**. Где есть пробел — **рекомендация по усилению** (лучше заложить в v1 как поля шапки или один блок «Сводка»).

### A. Типы мероприятий и сценарии работы

| Кейс | Статус / мяч | Смета | Файлы | Тайминг | Коммуникации | Заявки реквизита | Замечание |
|------|--------------|-------|-------|---------|--------------|------------------|-----------|
| Тимбилдинг, даты плавали 3 раза | ✅ лог + смена статуса | ✅ версии смет | ✅ договор/КП в Документация | ✅ перенос дней | ✅ по контакту | ✅ новая заявка при новых датах | История **дат мероприятия** в логе желательна явно (см. § «Усиления»). |
| Промо в ТЦ, много субов | `AWAITING_VENDOR` / мяч VENDOR | ✅ блоки тех/орг | ✅ акты субов в Документация | ✅ монтаж/слоты | ✅ «ждём площадку» в журнале | ✅ отдельная заявка под выезд | Ок. |
| «Разработайте с нуля», месяцы без даты | `INTERNAL_PREP` / `PROPOSAL_REVISION` | ✅ версии без события | ✅ бриф в Документация / Дизайн | ⚠️ дни без календарной даты | ✅ | ❌ реквизит до дат нельзя — ок по ТЗ | Тайминг: уже заложены **дни без обязательной даты** — использовать «День 1», «Репетиция» текстом. |
| Корпоратив + вечеринка (два дня) | `LIVE` / `WRAP_UP` | ✅ разделы смет | ✅ | ✅ два ScheduleDay | ✅ | ✅ две заявки или одна на диапазон | Одна заявка с диапазоном дат = одна строка в блоке заявок. |
| Онлайн-трансляция (без реквизита) | статусы продаж | ✅ только услуги | ✅ | ✅ сценарий эфира | ✅ | — | Ок без заявок. |
| Срочный «завтра событие» | `READY_TO_RUN` → `LIVE` | ✅ быстрая смета v1 | ✅ | ✅ один день | ✅ звонки в журнале | ✅ заявка из каталога | Ок. |
| Клиент пропал / не отвечает | `ON_HOLD` + мяч CLIENT | — | — | — | последняя запись «нет ответа» | — | **Напоминание** — в «Позже»; в v1 достаточно журнала + статус. |
| Несколько юрлиц (мы + суб счета) | `CONTRACT_SIGNED` | ✅ себестоимость / клиент | ✅ папки по типам | — | ✅ | — | Если нужно **раздельно хранить «наш договор» и «суб»** — подпапки в Документация или теги имён файлов; модель не мешает. |
| Параллельно: ждём деньги И правки сметы | один статус | ⚠️ | ✅ | — | ✅ | — | **Ограничение:** один статус не отражает два трека. Решение: в шапке **два чек-флага/подсказки** «Оплата: ждём» / «Документы: ждём» или свободное поле «Открытые блокеры» (см. § «Усиления»). |

### B. Документы и сметы

| Кейс | Покрытие | Пробел / усиление |
|------|----------|-------------------|
| КП v1, v2, v3 | `EstimateVersion` + PDF из конструктора | В карточке показывать **текущую версию + список версий** одной строкой. |
| КП уехало клиенту файлом из Excel | Файл в **Сметы** | Нельзя сравнить с конструктором автоматически — ок; в UI подпись «только файл». |
| НДС / без НДС в договоре | Документ + запись в журнале | При необходимости — поле **«Условия оплаты / НДС»** текстом в шапке. |
| Частичная предоплата / этапы | Журнал + файлы счетов | Для наглядности v1: поле **«Финансы (свободный текст)»** или чеклист в шапке. |

### C. Люди и коммуникации

| Кейс | Покрытие |
|------|----------|
| Один контакт | Один `ProjectCommunicationThread`. |
| Два контакта (закупка + маркетинг) | Два потока, понятно кто с кем говорил. |
| Суб звонил «с поля» | Запись в журнале от имени Wowstorg: «суб X сообщил …» — без отдельного пользователя-суба. |
| Смена контакта у клиента | Новый контакт + новый поток; старый архивируем/помечаем неактивным (**поле `isActive` на контакте** — желательно в модели). |

### D. Реквизит и склад

| Кейс | Покрытие |
|------|----------|
| Несколько выездов (монтаж + день Х) | Несколько `Order` с `projectId` — в карточке список с датами и статусами заказов (как сейчас в системе). |
| Дозаказ реквизита | Ещё одна заявка с того же проекта. |
| Отмена только реквизита, проект жив | `Order` → CANCELLED; проект не трогаем; лог проекта «заявка отменена». |

### E. Где структуры **мало** для «максимальной информативности»

Один **статус** + **мяч** не передают одновременно «и деньги ждём, и клиент молчит». Это не баг спеки — это сознательное упрощение.

**Рекомендуемые усиления карточки (v1, лёгкие поля):**

1. **`eventDateNote` + `eventDateConfirmed` (bool)** — черновая дата или диапазон текстом + флаг «дата подтверждена»; смены писать в `ProjectActivityLog`.  
2. **`openBlockers` (textarea)** — одна строка «что мешает прямо сейчас» (видно в шапке без чтения всего журнала).  
3. **`internalSummary` (textarea)** — «суть в двух предложениях» для коллеги, кто открыл карточку первый раз.  
4. **Контакт `isActive`** — не путать старых ЛПР с новыми.  
5. **Блок «Заявки»** в UI: таблица `id`, даты, статус заказа, ссылка на существующую карточку заказа — чтобы не дублировать логику склада.

**Позже (не блокер ясности v1):** деньги в цифрах (счёт/оплачено), напоминания, теги проектов, связь «родительский / дочерний» проект для этапов.

### Вывод

Для **2 пользователей** и описанного объёма текущей структуры **достаточно**, чтобы по карточке понимать стадию, переговоры, документы, сметы, сценарий и реквизит — при условии **собранного UI**: шапка (статус + мяч + клиент + ответственный + даты/блокеры/сводка) + вкладки/секции (коммуникации, файлы, сметы, тайминг, заявки, лог).  

Без полей **§ «Усиления»** часть кейсов потребует листать длинный журнал — не ошибка модели, а снижение скорости ориентации.
