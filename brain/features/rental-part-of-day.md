# Утро / вечер на границах периода аренды

## Цель

Дать пользователю явный выбор **части дня** на дате **начала** и **окончания** мероприятия, чтобы из дат считать **оплачиваемые дни аренды** без расхождения между каталогом и сервером.

## Границы (на старте)

- Резерв остатков (`reserve.ts`): по умолчанию **без** изменения семантики пересечения целых дат — только если бизнес отдельно не потребует иное.
- Issuance heatmap: без привязки к «полдням».
- Дробные дни (0.5) — только если явно утверждены в продукте; иначе целые дни по матрице.

## Затронуто

- **Prisma:** `Order` — два enum-поля (начало/конец «утро|вечер»).
- **Ядро:** `src/server/orders/order-pricing.ts` (+ общий расчёт в `src/lib/rental-days.ts` или рядом).
- **API:** `POST /api/orders`, `GET /api/orders/[id]`, при редактировании дат — `greenwich-edit` / `warehouse-edit`.
- **Сервер:** `create-order.ts`, снимки смет при необходимости, Telegram/XLSX при упоминании периода.
- **Клиент:** `catalogDates` + storage, `src/app/catalog/page.tsx`, `src/app/cart/page.tsx`, `src/app/orders/[id]/page.tsx`, при необходимости `ProjectEstimatePanel.tsx`.

## Инварианты

- Один источник формулы `billableRentalDays` для UI-превью и `calcOrderPricing`.
- Не менять ADR Serializable на инвентаре без отдельного решения.
- После добавления тел к существующим route — при необходимости `npm run brain:inventory`.

## Приёмка

- [ ] Матрица правил утверждена владельцем продукта (все сочетания + один день / несколько дней).
- [ ] Миграция с дефолтами для существующих `Order`; поведение старых заявок понятно.
- [ ] Каталог ↔ корзина ↔ итог `POST /api/orders` дают одинаковую сумму при тех же строках корзины.
- [ ] Редактирование дат и переключателей в разрешённых статусах согласовано с складом/Greenwich.
- [ ] Сметы/архивные тексты при необходимости показывают «с утра / до вечера» (или эквивалент).
- [ ] Компонент переключателя доступен (keyboard + screen reader).
