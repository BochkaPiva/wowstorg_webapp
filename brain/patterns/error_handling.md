# Паттерн: ошибки и ответы API

## Формат ошибки HTTP

Единый формат из `src/server/http.ts`:

```ts
return jsonError(status, "Сообщение для клиента", optionalDetails);
```

Тело: `{ error: { message, details } }`.

## Клиент (`use client`)

- После **`fetch`**: не вызывать **`res.json()`** без защиты — использовать **`readJsonSafe`** (`src/lib/fetchJson.ts`) или `.catch(() => null)`.
- Цепочки `.then(r => r.json())` без обработки отклонений → риск **`unhandledrejection`** и оверлея Next с `[object Event]`. См. [`docs/DEBUG_UNHANDLED_REJECTION.md`](../../docs/DEBUG_UNHANDLED_REJECTION.md).

## Сервер

- Ожидаемые доменные сбои (нет заказчика, нет позиции, превышен резерв) → **4xx** с ясным текстом.
- Неожиданные исключения могут дать **500**; для критичных путей лучше явно логировать и по возможности оборачивать.

## Уведомления

- Сбой Telegram **не должен** превращать успешное создание заявки в 500. Использовать отложенное выполнение и **логирование**, не пробрасывать в ответ пользователю как фатальное.
