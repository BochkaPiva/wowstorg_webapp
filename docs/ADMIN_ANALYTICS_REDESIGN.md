# Редизайн админ-аналитики

Дата актуализации: 2026-04-28

## Цель
Пересобрать страницу `Админка -> Аналитика` в управленческий центр с одним глобальным периодом, вкладками и экспортом в аккуратный XLSX.

Страница должна отвечать на два типа вопросов:

1. Сухая отчетность: сколько заявок, проектов, клиентов, денег, маржи, статусов, отмен и завершений.
2. Управленческие сигналы: где проект завис, где низкая маржа, где нет сметы, где нет активности, кто ценный клиент.

Главный риск фичи — неверные расчеты. Поэтому реализация должна начинаться с серверного read-model и единых helper-формул, а не с UI.

## Итоговая структура UI
Один глобальный период `from/to` сверху. Он применяется ко всем вкладкам и экспортам.

Вкладки:

1. `Обзор`
2. `Реквизит`
3. `Заказчики`
4. `Проекты`

Кнопки экспорта:

- `Скачать весь отчет` — один XLSX за глобальный период, листы разделены по смыслу.
- `Скачать реквизит`
- `Скачать заказчиков`
- `Скачать проекты`

Локальных date-фильтров внутри вкладок быть не должно.

## Текущие источники правды

### Реквизит / заявки
Код:

- `src/server/admin-analytics.ts`
- `src/server/orders/order-pricing.ts`
- `src/server/admin-analytics-xlsx.ts`
- `src/app/api/admin/analytics/route.ts`
- `src/app/api/admin/analytics/export/route.ts`

Текущая денежная логика реквизита:

- период фильтруется по `Order.endDate`;
- денежные KPI считаются по `Order.status = CLOSED`;
- rental revenue считается через `calcOrderPricing`;
- скидка на заявку учитывается только в rental subtotal;
- услуги `delivery/montage/demontage` не скидируются;
- для закрытых заявок используется `issuedQty ?? requestedQty`;
- выручка реквизита распределяется по строкам через `lineAllocations`;
- рентабельность позиции:
  - `purchaseCost = purchasePricePerUnit * total`
  - `grossProfit = itemRevenue - purchaseCost`
  - `paybackRatio = itemRevenue / purchaseCost`
  - `roiPercent = (itemRevenue - purchaseCost) / purchaseCost * 100`

Эту логику нельзя переписывать “по памяти”. Новая вкладка `Реквизит` должна переиспользовать эти расчеты.

### Проекты / сметы
Код:

- `src/server/projects/estimate-read-model.ts`
- `src/server/projects/estimate-xlsx.ts`
- `src/lib/project-estimate-totals.ts`
- `src/lib/project-estimate-requisite.ts`
- `src/lib/project-estimate-local-line.ts`
- `src/app/projects/[id]/ProjectEstimatePanel.tsx`

Текущая формула проектной сметы:

```text
clientSubtotal = сумма клиентских строк
internalSubtotal = сумма внутренних затрат
commission = round(clientSubtotal * 0.15)
revenueTotal = clientSubtotal + commission
tax = round(revenueTotal * 0.06)
grossMargin = revenueTotal - internalSubtotal
marginAfterTax = grossMargin - tax
marginAfterTaxPct = marginAfterTax / revenueTotal * 100
```

Константы:

- `PROJECT_ESTIMATE_COMMISSION_RATE = 0.15`
- `PROJECT_ESTIMATE_TAX_RATE = 0.06`

Строки сметы:

- `LOCAL` / `CONTRACTOR`: берутся из `ProjectEstimateLine`, `costClient` нормализуется через `normalizedLocalLineCostClientNumber`.
- `REQUISITE`: если секция связана с реальной заявкой, строки строятся через `calcOrderPricing`, то есть учитывают Greenwich multiplier и ручные скидки.
- `DRAFT_REQUISITE`: demo/draft-заявка без дат считается через `calcProjectEstimateRequisiteTotal`.
- услуги связанной заявки добавляются отдельными service rows с `costClient` и `costInternal`.

Для проектной аналитики нужно создать общий серверный helper/read-model, который считает проектные деньги тем же способом, что текущая проектная смета и XLSX. Нельзя делать отдельную ad-hoc формулу в `admin-analytics.ts`.

### ProjectActivityLog
Код:

- `src/server/projects/activity-log.ts`
- `src/app/api/projects/[id]/route.ts`
- `src/app/api/projects/route.ts`
- `prisma/schema.prisma`

Модель:

- `ProjectActivityLog.kind`
- `ProjectActivityLog.payload`
- `ProjectActivityLog.createdAt`

Смена статуса проекта фиксируется в `PROJECT_UPDATED` через `payload.changes.status`, когда карточка проекта редактируется.

Ограничение: отдельной таблицы истории статусов нет. Поэтому все метрики “время в статусах” являются расчетом по журналу событий и должны быть подписаны в UI.

## Период и дата-основания

### Глобальный период
Формат: `YYYY-MM-DD`.

Правила:

- `from` включительно;
- `to` включительно на уровне UI, на сервере безопаснее преобразовывать в end-exclusive `to + 1 day`;
- если `from > to`, API возвращает 400.

### Дата-основания по вкладкам

| Вкладка | Дата фильтра |
|---|---|
| `Реквизит` | `Order.endDate` |
| `Проекты` | `Project.createdAt` |
| `Заказчики` | проекты по `Project.createdAt`, заявки по `Order.endDate` в отдельных метриках |
| `Обзор` | смешанный источник, явно подписанный как `факт` и `прогноз` |

## Вкладка `Обзор`
Задача: дать картину бизнеса за 15 секунд.

### KPI
- Факт выручки по реквизиту.
- Факт выручки услуг.
- Факт валовой прибыли реквизита по tracked items.
- Закрытые заявки.
- Средний чек закрытой заявки.
- Прогнозная выручка проектов.
- Прогнозная маржа проектов после налога.
- Активные проекты.
- Завершенные проекты.
- Отмененные проекты.
- Проекты без активности 7+ дней.
- Проекты с низкой маржой.
- Повторные заказчики.

### Блоки
- KPI-полоса.
- `Что требует внимания`: stale-проекты, низкомаржинальные проекты, проекты без сметы, проекты без связанной заявки.
- `Факт vs прогноз`: факт по закрытым заявкам отдельно от прогноза по сметам.
- Мини-топы: топ клиентов, топ проектов, топ реквизита.

Задача: сохранить текущую сильную складскую аналитику и привести ее к новому UI.

### KPI
- Все заявки за период.
- Закрытые заявки.
- Выручка всего.
- Выручка реквизита.
- Выручка услуг.
- Средний чек.
- Средняя длительность аренды.
- Выручка по delivery/montage/demontage.

### Таблицы
- Топ реквизита по выдачам.
- Топ реквизита по выручке.
- Топ заказчиков по фактической сумме заявок.
- Рентабельность:
  - позиция;
  - тип;
  - количество;
  - закуп за единицу;
  - закуп всего;
  - выручка;
  - gross profit;
  - payback ratio;
  - ROI.

### Анти-топы
Можно добавить после базовой миграции:

- позиции с закупом и нулевой выручкой;
- позиции с отрицательным ROI;
- позиции с самой низкой окупаемостью.

## Вкладка `Проекты`
Задача: дать и сухую отчетность, и управленческие сигналы.

### Сухие KPI
- Всего проектов за период.
- Активные.
- Завершенные.
- Отмененные.
- Архивные.
- Доля завершения.
- Доля отмены.
- Проекты с основной сметой.
- Проекты без сметы.
- Проекты со связанной заявкой.
- Проекты без связанной заявки.
- Проекты с подтвержденной датой.
- Проекты без подтвержденной даты.
- Средняя прогнозная выручка проекта.
- Средняя прогнозная маржа после налога.
- Средняя маржинальность после налога.
- Среднее число заявок на проект.
- Среднее число версий сметы.

### Финансы
Считать по основной версии сметы только для неотмененных проектов. `CANCELLED` проекты остаются в операционных метриках отмен, воронке и conversion/cancel rate, но не входят в прогнозную выручку, маржу, средний чек проекта, топы по выручке/марже и финансовую таблицу.

- `clientSubtotal`
- `commission`
- `revenueTotal`
- `internalSubtotal`
- `tax`
- `grossMargin`
- `marginAfterTax`
- `marginAfterTaxPct`

Таблицы:

- топ прибыльных проектов;
- топ низкомаржинальных проектов;
- проекты с отрицательной/нулевой маржой;
- распределение проектов по диапазонам маржи:
  - `< 0%`
  - `0-15%`
  - `15-30%`
  - `30%+`

### Воронка проектов
Сухая воронка:

```text
созданы
-> есть основная смета
-> есть подтвержденные даты
-> есть связанная заявка
-> завершены
```

Важно: это не строгий state machine, а управленческий срез наличия обязательных артефактов.

### Где застреваем
Считать по `ProjectActivityLog`.

Метрики:

- среднее время в текущем статусе;
- проекты дольше 7/14 дней без активности;
- топ статусов с самым длинным ожиданием;
- среднее время от создания проекта до первой сметы;
- среднее время от создания проекта до первой связанной заявки;
- среднее время от создания проекта до завершения;
- среднее время от создания проекта до отмены.

UI обязательно показывает tooltip:

> Расчет по журналу активности проекта (`ProjectActivityLog`). Если статус менялся до появления записи в журнале или вручную нестандартным способом, метрика может быть приблизительной.

### Индекс здоровья проекта
Это не финансовая истина, а сигнал приоритизации.

Предлагаемая шкала 0-100:

- `-25`: нет активности 14+ дней;
- `-15`: нет активности 7+ дней;
- `-20`: активный проект без основной сметы;
- `-15`: активный проект без связанной заявки;
- `-10`: дата мероприятия не подтверждена;
- `-20`: маржа после налога ниже 15%;
- `-30`: маржа отрицательная;
- `-10`: проект дольше 14 дней в одном статусе;
- `+10`: есть смета, даты и связанная заявка.

Минимум 0, максимум 100.

Индекс нужен только для сортировки и блока “требует внимания”, не для бухгалтерской отчетности.

### Риски
- Stale 7+ дней.
- Stale 14+ дней.
- Нет основной сметы.
- Нет связанной заявки.
- Низкая маржа.
- Отрицательная маржа.
- Нет подтвержденных дат.
- Дата близко, а статус не готов к проведению.

### Требования к оформлению Excel
- Заголовок отчета на каждом листе.
- Период отчета.
- Единый стиль шапок.
- Закрепленная шапка таблиц.
- Денежные колонки с форматом `#,##0`.
- Проценты с форматом `0.00%`.
- Автоширина или фиксированные широкие колонки.
- Лист `Методология` в общем отчете обязателен.

### Методология в Excel
На листе `Методология` указать:

- `Реквизит`: факт по закрытым заявкам, период по `Order.endDate`.
- `Проекты`: прогноз по основной версии сметы, период по `Project.createdAt`.
- `Заказчики`: проектные метрики по проектам, фактические заявки отдельно.
- `Проектная комиссия`: 15%.
- `Условный налог`: 6%.
- `Статусы`: расчет по `ProjectActivityLog`.
- `Скидки`: в факте реквизита учитываются через `calcOrderPricing`, услуги не скидируются.

## Предлагаемый серверный read-model

Расширить `src/server/admin-analytics.ts` или разбить рядом на внутренние helper-файлы:

- `getAdminAnalyticsData(scope)` — главный фасад.
- `getRequisiteAnalytics(scope)` — текущая логика реквизита.
- `getProjectAnalytics(scope)` — проектные KPI, финансы, риски.
- `getCustomerAnalytics(scope)` — клиентские метрики.
- `buildProjectFinancialSnapshot(project)` — единая формула проектных денег.
- `buildProjectActivityMetrics(project)` — статусы и stale.
- `buildAnalyticsExport(data, section)` — общий XLSX builder.

Внешний API может остаться:

- `GET /api/admin/analytics?from=&to=`
- `GET /api/admin/analytics/export?section=global|requisites|projects|customers&from=&to=`

Но payload нужно версионировать структурно:

```ts
{
  period,
  overview,
  requisites,
  projects,
  customers,
  methodology
}
```

## Риски внедрения

### Неверные деньги по проектам
Причина: повторение формул вручную.

Защита:

- один helper проектных финансов;
- сверка с `ProjectEstimatePanel`;
- сверка с проектным XLSX;
- тестовые проекты с LOCAL, CONTRACTOR, REQUISITE и DRAFT_REQUISITE.

### Сломать текущую аналитику реквизита
Причина: большой рефактор `admin-analytics.ts`.

Защита:

- сначала обернуть текущий результат во вкладку `requisites`;
- не менять формулы `calcOrderPricing`;
- regression snapshot по текущему `totalRevenue/itemsRevenue/servicesRevenue/profitability`.

### Тяжелые запросы
Причина: проекты + сметы + activity logs + orders за большой период.

Защита:

- ограниченные `select`;
- считать за период, не весь мир;
- не грузить файлы/контакты/расписание в аналитику;
- при необходимости добавить lightweight server cache только для read-only аналитики.

### Неполная история статусов
Причина: `ProjectActivityLog` не был отдельной status-history.

Защита:

- подписать приблизительность;
- считать только по известным `PROJECT_UPDATED.payload.changes.status`;
- для текущего статуса брать `lastStatusChangeAt ?? project.createdAt`.

### Перегруз UI
Причина: слишком много KPI.

Защита:

- в `Обзоре` только 8-12 ключевых карточек;
- подробные таблицы во вкладках;
- риски показывать как action-list, не как бесконечную таблицу.

## Regression checklist

### Реквизит
- [ ] Закрытая заявка без скидки совпадает со старой аналитикой.
- [ ] Закрытая заявка со скидкой уменьшает только rental revenue.
- [ ] Услуги не скидируются.
- [ ] Top revenue сходится с item line allocations.
- [ ] Profitability использует revenue после скидки.

### Проекты
- [ ] Проект без сметы попадает в `без сметы`.
- [ ] Проект с основной сметой считает финансы.
- [ ] REQUISITE-секция со связанной заявкой совпадает с проектной сметой.
- [ ] DRAFT_REQUISITE входит в прогноз.
- [ ] CONTRACTOR/LOCAL строки учитывают `costClient` и `costInternal`.
- [ ] Маржа после налога совпадает с `calcProjectEstimateTotals`.

### Заказчики
- [ ] Один клиент с несколькими проектами считается repeat.
- [ ] Отмененные проекты влияют на cancel rate.
- [ ] Завершенные проекты влияют на completion rate.
- [ ] LTV явно подписан как смешанный показатель.

### Excel
- [ ] Full export содержит все листы.
- [ ] Separate export содержит только нужный раздел.
- [ ] Деньги и проценты отформатированы.
- [ ] Методология присутствует.

## Что не делать в первой итерации
- Не добавлять новую таблицу истории статусов.
- Не строить графики через тяжелые библиотеки.
- Не делать прогноз будущих заявок по AI/эвристикам.
- Не менять проектную смету ради аналитики.
- Не вводить бухгалтерский учет налогов вместо текущего условного `6%`.
