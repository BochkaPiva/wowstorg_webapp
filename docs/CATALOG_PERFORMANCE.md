# Оптимизация каталога

Документ фиксирует первый этап ускорения каталога без ручной переработки уже загруженных фото.

## Что изменено

- Каталог перешёл на серверную пагинацию по `32` карточки на страницу.
- Поиск больше не дёргает API на каждый символ мгновенно: используется короткий debounce.
- Категории и пакеты загружаются отдельно от списка позиций, чтобы не перезапрашивать их при каждом изменении поиска и дат.
- Карточки каталога запрашивают облегчённое превью фото через `GET /api/inventory/positions/[id]/photo?w=480`.
- Полный оригинал изображения не удаляется и остаётся доступным для остальных сценариев.
- На странице каталога отключены анимированные фоновые orb-эффекты и снят тяжёлый blur в оболочке.

## Что это даёт

- Меньше одновременно отрисованных карточек.
- Меньше байтов и меньше decode-нагрузки у изображений.
- Меньше GPU/compositing-нагрузки на слабых ПК.
- Более предсказуемую отзывчивость поиска и прокрутки.

## Что важно для фото

Ручная переработка старых фотографий не требуется.

Ускорение для уже существующих файлов достигается кодом:

- каталог просит уменьшенную версию через query-параметр `w`
- серверный route уменьшает изображение на лету и отдаёт WebP-превью
- исходный файл в storage остаётся без изменений

Это первый этап. Если после эксплуатации понадобится ещё большее ускорение, следующим шагом можно добавить отдельные stored thumbnails.

## Пагинация и совместимость

- Обычный каталог использует пагинацию.
- Специальный режим `ids=...` в `/api/catalog/items` сохранён без пагинации, чтобы не ломать корзину и внутренние загрузки по списку item id.

## Следующий этап при необходимости

- stored thumbnails для ещё меньшей нагрузки на сервер
- infinite scroll вместо кнопочной пагинации
- более агрессивная оптимизация резервов в `/api/catalog/items`, если узким местом останется серверный расчёт доступности
