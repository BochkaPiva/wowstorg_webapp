# Операционный дашборд Wowstorg

## Цель

Главная страница Wowstorg должна быть рабочим экраном на ближайшие дни: что сделать сегодня, что скоро станет проблемой, какие проекты, заявки и задачи требуют действия. Дашборд не должен быть справочником по всем данным системы и не должен дублировать очевидные пояснения под заголовками.

## Принципы UI

1. **Первый экран - только действия и риски.** Если элемент не помогает принять решение сегодня или в ближайшие дни, он уходит ниже или в раскрываемый блок.
2. **Мало поясняющего текста.** Заголовок блока должен быть самодостаточным: `Сегодня`, `Ближайшие дни`, `Сигналы`, `Мои задачи`, `Заявки`, `Склад`.
3. **Компактность важнее полноты.** На первом экране показываем ограниченное количество карточек, остальное доступно через ссылки `Все задачи`, `Все проекты`, `Все заявки`.
4. **События из разных сущностей собираются в одну временную картину.** Задача, заявка и проект могут жить в одной ленте дня, если для пользователя это одно рабочее расписание.
5. **Сигналы не должны висеть из-за ерунды.** Нет срока, риска денег или риска операционного срыва - нет сигнала.
6. **Календарь загрузки остается, но по умолчанию свернут.** На главной виден компактный статус, годовой heatmap раскрывается по требованию.

## Первый Экран

### 1. Сегодня

Крупный верхний блок, 3-5 элементов максимум.

Показывает:
- просроченные задачи текущего пользователя;
- задачи текущего пользователя с дедлайном сегодня;
- заявки, где сегодня готовность, сборка, выдача, окончание аренды или приемка;
- проекты с датой мероприятия сегодня;
- критические сигналы.

Пустое состояние: `Сегодня спокойно`.

### 2. Ближайшие дни

Лента на 5 дней: `Сегодня`, `Завтра`, `Послезавтра`, далее даты.

В каждой колонке:
- задачи с дедлайном;
- заявки по ключевым датам;
- даты мероприятий проектов;
- важные проектные сигналы.

Карточки должны быть маленькими: тип, название, время/дата, привязка к проекту или заявке. Без длинных описаний.

### 3. Мои задачи

Группы:
- `Просрочено`;
- `Сегодня`;
- `Скоро`;
- `Без срока`.

В карточке:
- название;
- статус/колонка;
- дедлайн;
- проект или заявка, если есть;
- прогресс чеклиста;
- быстрый переход на задачу/доску.

Не показывать длинное описание задачи на дашборде.

### 4. Сигналы

Сигнал - это не подсказка и не заполненность карточки. Это ситуация, где нужно действие или осознанное ожидание.

Каждый сигнал:
- имеет тип;
- имеет серьезность `info | warning | critical`;
- имеет конкретную причину;
- имеет ссылку на место действия;
- может быть отложен, если это не критический дедлайн сегодня.

## Сигналы: Разрешенные Сценарии

### TASK_OVERDUE

**Когда показывать:**
- `WorkTask.completedAt IS NULL`;
- `dueDate < today`;
- задача назначена текущему пользователю или относится к активному проекту.

**Серьезность:**
- `critical`, если просрочка 2+ дня;
- `warning`, если просрочка 1 день.

**Когда не показывать:**
- задача завершена;
- задача без дедлайна;
- проект закрыт/отменен и задача скрыта из основного YouGile.

### TASK_DUE_SOON

**Когда показывать:**
- задача не завершена;
- дедлайн сегодня или завтра;
- задача назначена текущему пользователю.

**Серьезность:** `warning`.

**Когда не показывать:**
- дедлайн позже завтрашнего дня;
- задача не назначена текущему пользователю;
- задача завершена.

### PROJECT_EVENT_SOON_WITHOUT_ESTIMATE

**Когда показывать:**
- проект активен: не `COMPLETED`, не `CANCELLED`, `archivedAt IS NULL`;
- дата мероприятия подтверждена;
- `eventStartDate` в ближайшие 14 дней;
- нет основной версии сметы.

**Серьезность:**
- `critical`, если до мероприятия 3 дня или меньше;
- `warning`, если 4-14 дней.

**Когда не показывать:**
- дата не подтверждена;
- проект дальше 14 дней;
- есть основная версия сметы.

### PROJECT_EVENT_SOON_WITHOUT_OPEN_TASKS

**Когда показывать:**
- проект активен;
- дата мероприятия подтверждена;
- `eventStartDate` в ближайшие 7 дней;
- нет ни одной незавершенной задачи по проекту.

**Серьезность:** `info`, не красный сигнал.

**Когда не показывать:**
- дата дальше 7 дней;
- есть открытые задачи;
- проект завершен/отменен.

Этот сигнал спорный, поэтому в первой реализации можно оставить выключенным или показывать ниже критических сигналов.

### PROJECT_BLOCKED

**Когда показывать:**
- проект активен;
- `openBlockers` не пустой;
- сигнал не отложен.

**Серьезность:**
- `warning`;
- `critical`, если дата мероприятия в ближайшие 3 дня.

**Когда не показывать:**
- блокер пустой;
- проект закрыт/отменен;
- пользователь отложил сигнал и `muteUntil` еще не прошел.

### ORDER_STALE

**Когда показывать:**
- заявка активна;
- статус не менялся или заявка не обновлялась 3+ дня;
- `readyByDate` или `startDate` в ближайшие 7 дней;
- статус не финальный.

**Серьезность:**
- `warning`;
- `critical`, если готовность сегодня/завтра.

**Когда не показывать:**
- заявка далеко по датам;
- заявка закрыта/отменена;
- заявка недавно обновлялась.

### WAREHOUSE_SHORTAGE_SOON

**Когда показывать:**
- по заявкам ближайших 7 дней есть позиция, где доступное количество на дату меньше требуемого;
- учитываются текущие резервы и состояния `inRepair`, `broken`, `missing`.

**Серьезность:** `critical`.

**Когда не показывать:**
- дефицита нет;
- заявка отменена/закрыта.

Это лучше реализовывать после базового дашборда, чтобы не смешивать большой расчет доступности с первой итерацией UI.

## Что Не Является Сигналом

Не показывать на дашборде:
- отсутствие ответственного, если система ставит его автоматически;
- отсутствие описания проекта;
- отсутствие контактов;
- отсутствие файлов;
- отсутствие задач у проекта без близкой даты;
- отсутствие сметы у проекта без подтвержденной близкой даты;
- любые поля, которые просто желательно заполнить, но нет срока и риска.

## Отложить Сигнал

Использовать существующую модель `ProjectNotificationCooldown` там, где сигнал относится к проекту.

Правила:
- базовое отложение: 7 дней;
- максимальное ручное отложение: 30 дней;
- если проблема решена, сигнал пропадает независимо от отложения;
- если ситуация стала критической сегодня, отложенный `warning` может вернуться как `critical`.

Для задач отложение на первой итерации не нужно: задача либо закрывается, либо переносится на доске.

## Единый Тип Событий

Для UI дашборда лучше собрать нормализованный список событий.

```ts
type DashboardEventKind =
  | "task_due"
  | "task_overdue"
  | "order_ready"
  | "order_start"
  | "order_end"
  | "project_event"
  | "project_signal"
  | "warehouse_signal";

type DashboardEvent = {
  id: string;
  kind: DashboardEventKind;
  title: string;
  subtitle?: string;
  date: string;
  urgency: "normal" | "soon" | "today" | "overdue" | "critical";
  href: string;
  projectId?: string;
  orderId?: string;
  taskId?: string;
};
```

Рекомендуемый payload:

```ts
type OperationsDashboardPayload = {
  today: DashboardEvent[];
  upcomingDays: Array<{
    date: string;
    label: string;
    events: DashboardEvent[];
  }>;
  myTasks: {
    overdue: DashboardTask[];
    today: DashboardTask[];
    soon: DashboardTask[];
    noDueDate: DashboardTask[];
  };
  signals: DashboardSignal[];
  orders: ExistingOrdersSummary;
  warehouse: ExistingWarehouseSummary;
  issuanceCalendarSummary: {
    state: "calm" | "busy" | "peak";
    busyDaysNext30: number;
    nearestPeakDate: string | null;
  };
};
```

## API План

Рекомендуемый путь: новый `GET /api/dashboard/wowstorg/operations`.

Почему не расширять только текущий `/api/dashboard/wowstorg`:
- существующий route уже большой;
- новый контракт проще тестировать;
- можно оставить старые блоки на переходный период.

## UI План

1. `OperationsHero`
   - без подзаголовка-объяснения;
   - короткие счетчики: `Сегодня`, `Просрочено`, `Сигналы`, `Ближайшая заявка`.

2. `TodayStrip`
   - 3-5 главных событий;
   - если событий больше, кнопка `Еще N`.

3. `UpcomingDays`
   - 5 дней;
   - компактные карточки;
   - на мобильном горизонтальный скролл.

4. `MyTasksCompact`
   - группы по срокам;
   - максимум 8 задач на первом экране.

5. `SignalsPanel`
   - максимум 5 сигналов;
   - сортировка: `critical`, `warning`, `info`;
   - кнопка отложить только там, где это допустимо.

6. `IssuanceCalendarCollapsed`
   - по умолчанию строка-сводка;
   - кнопка `Развернуть`;
   - существующий `IssuanceCalendar` внутри.

7. `OrdersAndWarehouse`
   - ниже первого экрана;
   - сохраняет степпер ближайшей заявки;
   - склад остается компактной правой колонкой.

## Визуальный Тон

- меньше больших описательных абзацев;
- меньше вложенных карточек;
- карточки событий с понятными цветами типа, но без радуги;
- красный использовать только для настоящих просрочек/дефицита;
- фиолетовый оставить как основной брендовый акцент;
- календарь и аналитика не должны конкурировать с действиями на сегодня.

## Этапы Реализации

### Этап 1. Контракт и данные

- Создать серверный сборщик `src/server/dashboard/operations.ts`.
- Создать API `GET /api/dashboard/wowstorg/operations`.
- Собрать `today`, `upcomingDays`, `myTasks`, `signals`.
- Не менять схему БД.
- Переиспользовать существующие модели задач, проектов, заявок и `ProjectNotificationCooldown`.

### Этап 2. Новый первый экран

- Пересобрать Wowstorg-блок на `/home`.
- Убрать шумные подзаголовки.
- Вынести heatmap в свернутый блок.
- Оставить текущие заявки и склад ниже, чтобы не потерять рабочие сценарии.

### Этап 3. Сигналы

- Заменить текущую широкую логику `projectAttention` на строгие сценарии из этого документа.
- Добавить сортировку и лимиты.
- Добавить отложение для проектных сигналов.

### Этап 4. Полировка

- Проверить desktop/mobile.
- Проверить пустые состояния.
- Проверить, что первый экран не превышает разумную высоту.
- Прогнать `npm run build`.

## Приемка

- [ ] На первом экране видны события сегодня и ближайшие дни.
- [ ] Календарь загрузки по умолчанию свернут.
- [ ] Нет длинных поясняющих подзаголовков под очевидными заголовками.
- [ ] Сигналы появляются только по правилам из этого документа.
- [ ] Отложенный проектный сигнал пропадает до `muteUntil`.
- [ ] Задачи показываются по срокам, а не просто списком.
- [ ] Заявки и склад не исчезли, но стали вторичными по отношению к плану дня.
- [ ] На мобильном первый экран не превращается в длинную простыню.
- [ ] `npm run build` проходит.
