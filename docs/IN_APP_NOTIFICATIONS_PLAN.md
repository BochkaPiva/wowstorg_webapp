# Центр внутренних уведомлений

Дата: 2026-04-28

## Цель
Добавить полноценную систему внутренних уведомлений в интерфейсе сайта:

- колокольчик в верхнем меню;
- счетчик непрочитанных;
- выпадающая панель со списком;
- отметка прочитанного;
- очистка уведомлений;
- сохранение ачивок Greenwich;
- короткие уведомления по тем же важным событиям, что и Telegram.

## Текущая ситуация
В проекте уже есть основа:

- Prisma-модель `InAppNotification`;
- API `GET/PATCH /api/me/notifications`;
- компонент `src/app/_ui/InAppNotifications.tsx`;
- polling раз в 5 секунд;
- всплывающие toast-уведомления;
- ачивки Greenwich создают `InAppNotification`;
- часть статусов заявок уже создает in-app уведомления.

Проблемы:

- нет списка уведомлений;
- toast сразу помечает уведомление прочитанным;
- нет колокольчика и счетчика;
- нет очистки;
- enum `InAppNotificationType` содержит только `ACHIEVEMENT_UNLOCK`;
- уведомления статусов заявки записываются как `ACHIEVEMENT_UNLOCK`, что мешает развитию.

## Решение в этой итерации

### Модель
Расширить `InAppNotificationType`:

- `ACHIEVEMENT_UNLOCK`
- `ORDER_STATUS_CHANGED`
- `ORDER_UPDATED`
- `ORDER_DISCOUNT`
- `PROJECT_UPDATED`
- `SYSTEM`

`InAppNotification` остается главным источником правды для UI. Новые поля в этой итерации не добавляем, чтобы не усложнять схему.

### API
Расширить `/api/me/notifications`:

- `GET` — список + `unreadCount`;
- `PATCH` — отметить выбранные или все как прочитанные;
- `DELETE` — очистить выбранные или все уведомления текущего пользователя.

### UI
В `AppShell`:

- колокольчик рядом с кнопкой выхода;
- badge с числом непрочитанных;
- dropdown/panel со списком последних уведомлений;
- действия `Прочитать все`, `Очистить`;
- ссылки на заявку/проект из payload.

Toast остается, но больше не является единственным способом увидеть уведомление.

### События
Минимум:

- ачивка Greenwich;
- статус заявки изменился;
- смета отправлена;
- сборка началась;
- выдача;
- закрытие;
- отмена;
- запрос скидки;
- скидка применена.

Для сайта текст короче Telegram:

- title — суть;
- body — заказчик/контекст;
- payload — техническая ссылка.

## Browser push
Browser push работает как дополнительный канал поверх внутренних уведомлений.

Архитектура:

1. Любое событие создает `InAppNotification`.
2. Если у пользователя есть активные browser push subscriptions, `src/server/notifications/browser-push.ts` отправляет короткий push по этому же notification.
3. При клике на push открывается URL из `payloadJson`.

Используется:

- Service Worker `public/browser-push-sw.js`;
- таблица `BrowserPushSubscription`;
- env `VAPID_PUBLIC_KEY`, `VAPID_PRIVATE_KEY`, `VAPID_SUBJECT`;
- библиотека `web-push`;
- API `GET/POST/DELETE /api/me/push-subscriptions`.

Отправка best-effort: ошибка push не должна ломать создание in-app уведомления или бизнес-операцию. Подписки с HTTP 404/410 автоматически помечаются `disabledAt`.

## Риски

- Не создавать уведомления внутри критических транзакций как обязательный шаг.
- Не превращать список уведомлений в тяжелый запрос: нужен `limit`.
- Не удалять чужие уведомления: все операции фильтруются по `auth.user.id`.
- Не ломать ачивки: `ACHIEVEMENT_UNLOCK` остается.
- Не использовать browser push как единственный источник истории: история всегда в `InAppNotification`.
