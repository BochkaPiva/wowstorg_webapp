# Учет рентабельности реквизита (план внедрения)

## Цель

Добавить безопасный учет экономики по позициям инвентаря:
- цена закупа за 1 единицу (`purchasePricePerUnit`);
- расчет выручки и окупаемости по позициям;
- аналитика только для тех позиций, где цена закупа известна;
- доступ только для `WOWSTORG` (склад/админы склада).

## Текущее состояние (по коду)

1. `Item` хранит:
   - `pricePerDay`, `total`, `inRepair`, `broken`, `missing`, `internalOnly`, и т.д.
   - цены закупа нет.
2. CRUD позиций:
   - `POST /api/inventory/positions`
   - `PATCH /api/inventory/positions/[id]`
   - страницы `inventory/positions/new` и `inventory/positions/[id]`.
3. Аналитика:
   - `GET /api/admin/analytics` считает выручку по `CLOSED` заказам;
   - топ по выдачам/выручке и топ заказчиков;
   - срезов по периодам и XLSX-выгрузки нет.

## Зафиксированные продуктовые требования

1. Цена закупа хранится как одна сумма, без деления на НДС/без НДС.
2. Цена закупа задается за **1 единицу** позиции.
3. Поле необязательное (`nullable`), заполняется когда цена известна.
4. Рентабельность считается по выполненным (`CLOSED`) заявкам.
5. Доступ к полю и аналитике — только `WOWSTORG`.
6. В будущем желательна поддержка нескольких закупок одной позиции по разным ценам.
7. Выгрузка — в `xlsx`.

## Нерешенный функциональный момент (важно)

Для периодных срезов нужно выбрать «дату признания выручки»:

- **Вариант A (рекомендуется):** дата закрытия заявки (`closedAt`), т.е. факт завершения цикла.
- Вариант B: `endDate` аренды.
- Вариант C: дата выдачи.

### Рекомендация

Внедрить поле `Order.closedAt` и использовать его в аналитике периодов:
- при `check-in` выставлять `closedAt = now()`;
- для старых закрытых заказов сделать мягкий backfill (`updatedAt`).

Это даст корректный и объяснимый периодный учет.

## Этап 1 (безопасная база): одно поле закупа + базовая рентабельность

### Изменения схемы

В `Item`:
- `purchasePricePerUnit Decimal(12,2)?` (nullable)

Опционально на этом же этапе:
- `profitabilityTrackingEnabled Boolean @default(false)`  
  (если понадобится явно включать учет не по всем позициям с ценой).

### Изменения API

#### `POST /api/inventory/positions`
- принять `purchasePricePerUnit?: number | null`;
- валидация: `>= 0`;
- отсутствие поля = `null`.

#### `PATCH /api/inventory/positions/[id]`
- принять `purchasePricePerUnit?: number | null`;
- разрешить очищать значение (`null`).

### Изменения UI (только WOWSTORG)

1. `inventory/positions/new`:
   - добавить необязательное поле:
     - «Цена закупа за единицу (₽)»;
     - подсказка: «Заполняйте, если известна фактическая себестоимость единицы».
2. `inventory/positions/[id]`:
   - добавить поле в форму редактирования;
   - показывать расчет «Оценка закупа партии» = `purchasePricePerUnit * total` (если цена задана).

### Совместимость и риски

- Нулевой риск для старых данных: `null` не ломает существующий код.
- Если цена не заполнена, позиция не участвует в рентабельности.
- Текущие сценарии выдачи/возврата/каталога не затрагиваются.

## Этап 2: аналитика рентабельности (базовая)

### Расчетные метрики по позиции

Для каждой позиции с заданным `purchasePricePerUnit`:

1. **Выручка позиции (`revenue`)**
   - сумма по `CLOSED` заказам:
   - `line.pricePerDaySnapshot * line.requestedQty * rentalDays * payMultiplier`
   - (доп. услуги в выручку позиции не распределяются)

2. **Оценка закупа (`purchaseCost`)**
   - `purchasePricePerUnit * item.total`
   - для MVP берется текущее `total` (это приближенная оценка).

3. **Маржинальный результат (`grossProfit`)**
   - `revenue - purchaseCost`

4. **Окупаемость (`paybackRatio`)**
   - `revenue / purchaseCost` (если `purchaseCost > 0`)

5. **ROI (`roiPercent`)**
   - `((revenue - purchaseCost) / purchaseCost) * 100` (если `purchaseCost > 0`)

### API аналитики

Добавить отдельный endpoint (рекомендуется):
- `GET /api/admin/analytics/profitability`

Параметры:
- `from` (date-only, optional)
- `to` (date-only, optional)
- `itemType` (optional)
- `internalOnly` (optional)
- `q` (поиск по позиции, optional)

Возврат:
- агрегированные KPI;
- список позиций с метриками;
- сортировки по `revenue`, `roiPercent`, `paybackRatio`.

### Права

- только роль `WOWSTORG`.
- в публичные/greenwich-экраны эти данные не выводятся.

## Этап 3: срезы по датам и XLSX

### Дата среза

После внедрения `closedAt`:
- фильтр периода применять по `closedAt`.

### XLSX-выгрузка

Добавить endpoint:
- `GET /api/admin/analytics/profitability/export.xlsx?...`

Листы:
1. `Summary` — общие KPI.
2. `Items` — таблица по позициям:
   - item id/name/type
   - purchase price per unit
   - total qty
   - purchase cost
   - revenue in period
   - gross profit
   - payback ratio
   - ROI %

## Этап 4 (расширение): несколько закупок одной позиции

Текущая модель «одна цена закупа» не идеально покрывает докупки по другой цене.
Для корректной долгосрочной экономики нужен отдельный журнал закупок.

### Предлагаемая модель (будущее)

Новая таблица `ItemPurchaseBatch`:
- `id`
- `itemId`
- `qty`
- `unitCost`
- `purchasedAt`
- `comment`
- `createdById`

Тогда:
- `purchaseCost` считается по сумме батчей;
- можно считать точную средневзвешенную себестоимость;
- историчность не теряется.

### Оценка сложности

- MVP с одним полем: низко-средняя.
- Бэтчи закупок: средняя (доп. CRUD, валидация, перерасчет аналитики).

## Тест-план (обязательный минимум)

1. Создание позиции:
   - без закупочной цены;
   - с закупочной ценой;
   - с `0`.
2. Редактирование позиции:
   - установка/изменение/очистка закупочной цены.
3. Проверка, что каталог/заказы/выдача/приемка не изменились.
4. Аналитика:
   - позиция без цены не попадает в отчет рентабельности;
   - позиция с ценой попадает корректно;
   - фильтры периода работают.
5. Доступ:
   - Greenwich не видит закупочную цену и profitability-API.

---

## Порядок безопасного внедрения

1. Миграция БД (nullable поле) + backend валидация.
2. UI CRUD поля закупа (WOWSTORG only).
3. Базовый profitability API и простая таблица в аналитике.
4. Периоды + `closedAt` + XLSX экспорт.
5. (Опционально) переход на модель batch-закупок.

---

## Резюме

Рекомендуется начать с MVP:
- `purchasePricePerUnit` в `Item` (nullable),
- новый profitability endpoint только по `CLOSED` заказам,
- видимость только WOWSTORG,
- XLSX на втором шаге.

Это минимально рискованно, не ломает текущий функционал и дает быструю управленческую ценность.
