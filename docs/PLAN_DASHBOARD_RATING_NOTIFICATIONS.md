# План: Дашборд, рейтинг Greenwich, напоминания и календарь

Документ описывает нововведения после MVP: дашборды для Greenwich и Wowstorg, 100-балльный рейтинг сотрудников Greenwich, Telegram-напоминания и календарь активности. Все изменения вносятся без поломки текущего функционала.

---

## Зависимости и границы (архитектура)

- **Роли:** `GREENWICH` и `WOWSTORG` (модель `User`, поле `role`).
- **Заявки:** `Order` с `greenwichUserId` (опционально), `status`, `readyByDate`, `startDate`, `endDate`. Рейтинг начисляется только по заявкам с заполненным `greenwichUserId`.
- **Декларация возврата:** API `POST /api/orders/[id]/return-declared` переводит заказ в `RETURN_DECLARED` и создаёт `ReturnSplit` (phase `DECLARED`). Компонент рейтинга “своевременность” пересчитывается здесь по дате отправки на приёмку Greenwich.
- **Приёмка:** API `POST /api/orders/[id]/check-in` создаёт `ReturnSplit` (phase `CHECKED_IN`) и переводит заказ в `CLOSED`. Здесь пересчитывается компонент рейтинга “инциденты” (сломано/потеряно) по факту склада.
- **Позиции:** `Item` с `type` (ASSET, BULK, **CONSUMABLE**). Расходники (CONSUMABLE) не штрафуют рейтинг при поломке/потере/просрочке.
- **Главная:** `src/app/home/page.tsx` — единая точка входа; разный контент по `user.role` (уже есть разветвление по `isWowstorg`).
- **Очередь склада:** `GET /api/warehouse/queue` — список заявок в статусах до CLOSED; для дашборда можно переиспользовать логику «ближайшая по дате».
- **Мои заявки Greenwich:** `GET /api/orders/my` — заявки текущего пользователя; для дашборда — фильтр по статусам и сортировка по датам.
- **Уведомления:** `src/server/notifications/order-notifications.ts`, Telegram через `src/server/telegram.ts`. Напоминания — новый слой (cron/задачи), без изменения существующих API заявок.
- **Часовой пояс:** расчёты “по дням” и отправка напоминаний — по времени **Омска** (`Asia/Omsk`, UTC+6).
- **Ассет:** динозаврик уже есть — `public/dino.png`.

## 1. Рейтинг Greenwich (100 баллов)

### 1.1. Правила

| Событие | Изменение рейтинга |
|--------|---------------------|
| Стартовое значение для всех | **100** |
| Базовый бонус за приёмку возврата (check-in) | **+10** |
| Сломанная позиция (не расходник) | **−1** за каждую единицу |
| Потерянная позиция (не расходник) | **−3** за каждую единицу |
| Просрочка возврата | **−7** за каждый день просрочки |

- **Расходники (CONSUMABLE):** при поломке/потере по позиции с `Item.type === "CONSUMABLE"` баллы не снимаются (штрафы за BROKEN/NEEDS_REPAIR/MISSING по этим позициям = 0). Компонент просрочки по времени (OVERDUE) считается по факту отправки `return-declared`, даже если среди позиций есть расходники.
- Рейтинг не уходит ниже **0** (при расчёте ограничить: `newScore = max(0, current + delta)`).
- Компоненты пересчитываются раздельно:
  - “Своевременность” — при `return-declared` (когда Greenwich отправил возврат на приёмку).
  - “Инциденты” — при `check-in` (когда склад подтвердил фактические статусы).

### 1.2. Определения для расчёта

- **Дедлайн возврата:** `Order.endDate + 1 день` (включительно). Если мероприятие до 23 числа, а возврат отправили 24 — штраф не даётся.
- **Дата отправки на приёмку (Greenwich):** дата создания `ReturnSplit` с `phase = "DECLARED"` для заказа (берём “дата-only” по времени создания этих сплитов; рекомендуется хранить `returnDeclaredAt` или вычислять).
- **Просрочка (дней):** `max(0, diffDays(returnDeclaredDateOmsk, endDatePlus1Omsk))`, где `diffDays(a, b)` — сколько полных дней прошло между датами в часовом поясе Омска (если `a` <= `endDatePlus1`, то 0).
- **Как считается изменение рейтинга за заявку:**
  - На `return-declared` считается компонент `OVERDUE`: `-7 * overdueDays` по `DECLARED` и дедлайну `endDate + 1` (включительно).
  - На `check-in` считается компонент `INCIDENTS`: `+10 - brokenQty - 3*lostQty`, где `brokenQty` = сумма `NEEDS_REPAIR + BROKEN` по **не-расходникам**, `lostQty` = сумма `MISSING` по **не-расходникам**. По `Item.type === "CONSUMABLE"` штрафы = 0.
  - Итог за заявку = `OVERDUE + INCIDENTS`; итоговый рейтинг = `min(100, max(0, 100 + сумма изменения по всем заявкам))` (пересчитывается по сохранённым компонентам).

### 1.3. Хранение и API

- **Модель в БД (Prisma):**
  - `GreenwichRating` (или поле на `User`): `userId`, `score` (Int, текущий балл), `updatedAt`.
-  (без пользовательской истории “начисления/списания”; по желанию позже можно добавить аудит-таблицы).

- **Старт:** новым сотрудникам по умолчанию выдаём **100** (если записи рейтинга нет).

- **Перерасчёт (двухфазный):**
  - В `return-declared` после успешного `tx.order.update(..., status: "RETURN_DECLARED")`:
    - Если у заказа есть `greenwichUserId`, вычислить компонент `OVERDUE` по `ReturnSplit.phase="DECLARED"` и `Order.endDate+1`.
    - Обновить `GreenwichRating.score` (и поставить минимальный флаг/поле идемпотентности, чтобы повторные попытки не умножали штраф).
  - В `check-in` после успешного `tx.order.update(..., status: "CLOSED")`:
    - По `ReturnSplit.phase="CHECKED_IN"` определить `BROKEN/LOST` (и `OK`, если “всё в норме”).
    - Применить инциденты (но `OVERDUE` уже рассчитан по фазе `DECLARED`, а не по времени склада).

- **Дата отправки на приёмку:** для расчёта просрочки использовать дату `DECLARED` из `ReturnSplit` (createdAt сплитов). Для надёжности можно вычислять и сохранять `returnDeclaredAtDateOnly` на `Order` (или хранить в истории), но это необязательно для MVP.

### 1.4. Отображение

- **Главная Greenwich:** блок «Рейтинг» — интерактивный виджет: текущий балл (0–100), динозаврик (`/dino.png`) и фраза в зависимости от диапазона (см. ниже).
- **Заявки склада (очередь/карточка заявки):** рядом с именем сотрудника Greenwich показывать его текущий рейтинг (число или бейдж).

**Примеры фраз от динозаврика (по баллу):**
- 95–100: «Почти идеально! Так держать.»
- 80–94: «Хороший результат, ещё чуть-чуть до топа.»
- 50–79: «Есть куда расти. Возвращай вовремя и в порядке.»
- 1–49: «Рейтинг упал. Следующие заявки — шанс подняться.»
- 0: «Начни с аккуратных возвратов — рейтинг пойдёт вверх.»

## 2. Дашборд Greenwich (главная)

- **Блок «Заявки» (один блок):**
  - Плитки: **Активных заявок** (статусы, которые ещё не `CLOSED` и не `CANCELLED`), **Всего выполненных заявок** (`status="CLOSED"`, исключая `CANCELLED`).
  - Степпер статусов заявки + **карточка ближайшей по дате заявки** (как в очереди: даты, заказчик, сумма, статус). Всё в одном блоке.
  - При желании: краткий список «на сегодня / на завтра» (по readyByDate или startDate/endDate).

- **Блок «Рейтинг»:**
  - Оформление: интерактивно, с динозавриком и фразой по баллу.
  - Показать “за что изменился рейтинг” можно минимально (например одна последняя причина), но без полноценной истории MVP.

- **Зависимости:** `GET /api/orders/my` (расширить или добавить агрегаты для счётчиков и «ближайшей» заявки), API рейтинга (текущий балл и, опционально, минимальная причина последнего изменения).

## 3. Дашборд Wowstorg (главная)

- **Блок заявок:**
  - Количество **активных заявок** (статусы до CLOSED/CANCELLED).
  - Количество **выполненных заявок** (CLOSED, без отменённых).
  - **Карточка ближайшей по дате заявки** со степпером (как в очереди).

- **Блок «Реквизит»:**
  - Сводка: **сломано** / **потеряно** / **требует ремонта** только по **каталогу** (игнорировать складской реквизит `internalOnly=true`).
  - **Число позиций на складе:** количество уникальных `Item` (записей в таблице), удовлетворяющих тем же фильтрам (например активные, не только internalOnly). Это «позиций 2» (мячики + кубики), а не 70 единиц.
  - **Закончившиеся складские позиции:** список позиций с `internalOnly === true` и `total - inRepair - broken - missing === 0`. Блок показывать **только если такой список непустой**.

- **Зависимости:** данные из `Item` (агрегаты), список заявок из логики очереди; без изменения существующих API, только новые эндпоинты или расширение существующих.

## 4. Напоминания в Telegram

- **Время отправки:** ежедневно в **11:00** по Омску.
- **Склад:** за **1 день** до даты `readyByDate` отправлять напоминание: по заявке №… завтра нужно подготовить реквизит к дате `readyByDate` (или ближайшей ключевой дате); заказчик, при необходимости краткий список позиций.
- **Greenwich:** **в день возврата** (по `endDate`) — сегодня нужно вернуть реквизит по заявке №…; вовремя возврат — плюс к рейтингу, иначе возможен минус. Если `greenwichUserId` не назначен (внешняя заявка), уведомления отправляем в чат склада.
- Реализация: планировщик (cron или Vercel Cron, отдельный скрипт) раз в день обходит заказы по `readyByDate` / `endDate` и шлёт уведомления через существующий Telegram-слой. Не менять логику заявок и приёмки.
- Тон: дружелюбный, при желании с элементами геймификации (как обсуждалось).

## 5. Календарь активности (GitHub-style)

- Сетка квадратиков по дням за **весь год** (как GitHub), но клетки и палитра будут под наш дизайн (не “чёрным”).
- Цвет/яркость — от интенсивности активности в день (например количество заявок с событием в этот день: создание, выдача, приёмка).
- Сложность выше; реализовать после дашбордов и рейтинга. Источник данных: по `Order` (createdAt и/или даты смены статусов). При необходимости — использовать `Order.updatedAt` как прокси.

## 6. Принятые решения (MVP)

- **Дашборд Wowstorg — «в наличии» с арендой:** считаем занятость по активным заявкам через пересечение периода заказа с **текущим календарным днём по Омску** (как в `getReservedQtyByItemId`, но с `startDate = endDate = сегодня` в Омске).
- **Напоминания:** внешние заявки без `greenwichUserId` — в чат склада; ежедневная отправка в **11:00 по Омску** (реализация cron — следующий этап).

## 7. Порядок внедрения (без поломки текущего)

1. **Рейтинг (модель + логика):**
   - Миграция: таблица `GreenwichRating` (или поле на `User`), при необходимости поле/признак на `Order` для идемпотентности (чтобы повторные попытки не умножали штрафы).
   - В `return-declared`: после перевода в RETURN_DECLARED — расчёт/запись компонента `OVERDUE` по `DECLARED` и `endDate+1`.
   - В `check-in`: после перевода в CLOSED — расчёт/запись компонентов `BROKEN/LOST/PERFECT_RETURN` по `CHECKED_IN` (а `OVERDUE` не переопределять по времени склада).
   - API: GET текущий рейтинг пользователя (а “история” не нужна для MVP).

2. **Дашборд Greenwich:**
   - API для главной: счётчики заявок + ближайшая заявка (на базе данных, используемой в `/api/orders/my`).
   - Замена/расширение блока «Быстрые разделы» на `home/page.tsx`: блок заявок (плитки + степпер + карточка), блок рейтинга с динозавриком и фразой по баллу.

3. **Отображение рейтинга в заявках склада:**
   - В API очереди/архива при необходимости джойн с рейтингом по `greenwichUserId`; в UI очереди и карточки заявки — вывести рейтинг рядом с именем Greenwich.

4. **Дашборд Wowstorg:**
   - API: агрегаты по Item (сломано/потеряно/ремонт, число позиций, список закончившихся складских).
   - На главной для WOWSTORG: блок заявок (счётчики + ближайшая со степпером), блок реквизита.

5. **Напоминания:**
   - Скрипт/задача по расписанию, запросы к БД по датам, отправка через существующие Telegram-функции. Без изменений в API заявок.

6. **Календарь активности:** после стабилизации п. 1–5.

## 8. Риски и ограничения

- **Часовые пояса:** даты заявок и приёмки хранить в UTC, сравнения по «дню» делать в одной зоне (**Asia/Omsk**) для просрочки и напоминаний.
- **Производительность:** агрегаты по Item для дашборда — при большом числе позиций возможны индексы (например по `internalOnly`, `isActive`) и кэш при необходимости.
- **Рейтинг для старых заявок:** начисления/списания для истории рейтинга считаем только по событиям после внедрения (когда начнут происходить `return-declared`/`check-in`). Исторические заказы до этого не трогаем; стартовое значение 100 для новых/впервые увиденных сотрудников.
