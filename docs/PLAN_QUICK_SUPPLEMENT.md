# План: Быстрая доп.-выдача (Quick Supplement)

## Цель
Позволить добавлять **новые позиции** к уже выданной/идущей на выдачу заявке так, чтобы:
- ускорить оформление для Greenwich/Grinvich (и сценарий “нет интернета” через звонок на склад),
- не вмешиваться в уже работающий “основной” процесс заявки,
- не ломать учёт, приёмку и рейтинг Greenwich,
- обеспечивать понятный UI: быстрая доп.-заявка выглядит как отдельная заявка, но помечена и выделена.

## Роли
- **Grinvich / Greenwich сотрудник**: инициирует “быструю доп.-выдачу” из заявки в статусе `ISSUED`.
- **Wowstorg / Склад**: инициирует “быструю доп.-выдачу” из очереди заявок в статусе `ISSUED` и проводит выдачу по стандартной цепочке.

## Термины
- **Основная заявка (Parent)**: та заявка, к которой добавляются новые позиции.
- **Быстрая доп.-заявка (Quick Supplement)**: отдельная заявка, создаваемая поверх Parent.
- **Доп.-услуги**: `deliveryEnabled`, `montageEnabled`, `demontageEnabled` и соответствующие цены.

## Принятый вариант MVP
**Вариант B**: Quick Supplement — это **отдельная заявка**, связанная с Parent.

### Возврат/приёмка
- Quick Supplement **сдают на приёмку отдельно**, как отдельную заявку.
- “Общая приёмка” нескольких заявок **не делаем** (сильно снижает риск).

### Строгая совместимость со статусами (важно для бэкенда)
В текущей реализации ваш бэкенд “взводит” цепочку только по статусам:
- `start-picking` разрешён только при `Order.status === "APPROVED_BY_GREENWICH"`;
- `issue` разрешён только при `Order.status === "PICKING"`;
- `return-declared` разрешён только при `Order.status === "ISSUED"`;
- `check-in` разрешён только при `Order.status === "RETURN_DECLARED"`.

Поэтому Quick Supplement нельзя “создавать как обычную заявку через POST /api/orders”, иначе вы получите статус `SUBMITTED` и дальнейшие шаги не совпадут с интерфейсом.
Нужны отдельные endpoint(ы), которые создают Quick Supplement сразу в “стартовом” статусе из раздела “Контракт по статусам”.

## Ограничения MVP (что точно НЕ делаем)
- Quick Supplement **не использует доп.-услуги**.
- Quick Supplement **не редактируется по смыслу** (и UI это не должен позволять).
- Создание Quick Supplement возможно **только при Parent.status = `ISSUED`**.
- Quick Supplement не должен создавать дополнительных этапов “смета/согласование” (иначе теряется смысл “быстро”).

## Наследование данных
Quick Supplement должен наследовать (использовать значения Parent):
- `customerId` (заказчик),
- `startDate`, `endDate`, `readyByDate` (даты мероприятия),
- `eventName`, `comment` (если используются в шапке/сообщениях),
- `greenwichUserId` (тот же сотрудник, что и у Parent),
- любые другие “атрибуты мероприятия”, которые важны для одинаковости карточки и уведомлений.

Quick Supplement **не наследует доп.-услуги**:
- `deliveryEnabled = false`, `montageEnabled = false`, `demontageEnabled = false`,
- цены доп.-услуг отсутствуют/не используются.

> Примечание: в интерфейсе и уведомлениях доп.-услуги в Quick Supplement будут отсутствовать — это соответствует договорённости.

## Связь Parent ↔ Quick Supplement (обязательный контракт)
Quick Supplement должен содержать явную связь на Parent (для UI метки и для фильтрации “к какому заказу относится”).

Рекомендуемый контракт данных:
- в `Order` добавить nullable поле `parentOrderId` (или аналог),
- Quick Supplement = любой `Order`, у которого `parentOrderId` заполнен.

Дополнительно (для читаемости бизнес-логики):
- можно добавить флаг `isQuickSupplement`, но достаточно одной связи `parentOrderId`, если других “типов” заказов с parent не будет.

## Контракт по статусам (самое важное)
Статусы/роуты выдачи у вас жёстко привязаны к цепочке:
- `start-picking` разрешён только при `APPROVED_BY_GREENWICH`,
- `issue` разрешён только при `PICKING`,
- возврат (`return-declared`) разрешён только при `ISSUED`.

Следовательно, Quick Supplement должен создаваться **с корректным стартовым статусом**, который позволит следующему шагу выполняться “в текущем интерфейсе”.

### 1) Quick Supplement создаёт Greenwich/Grinvich
Сценарий:
1. Parent в статусе `ISSUED`.
2. Greenwich нажимает “быстрая доп.-выдача”.
3. Система создаёт Quick Supplement сразу в статусе `APPROVED_BY_GREENWICH`.
4. Wowstorg проверяет и нажимает “Начать сборку” (обычный `start-picking`), затем сразу “Выдать” (обычный `issue`).

Инвариант:
- Quick Supplement должен стартовать в `APPROVED_BY_GREENWICH`, иначе склад не сможет начать сборку стандартным роутом.

### 2) Quick Supplement создаёт склад (Wowstorg)
Сценарий:
1. Склад в очереди заявок нажимает “Быстрая доп.-выдача” у Parent в статусе `ISSUED`.
2. Склад формирует корзину (быстрый режим интерфейса).
3. Система создаёт Quick Supplement сразу в статусе `PICKING`.
4. После оформления склад нажимает “Выдать” (обычный `issue`).

Инвариант:
- Quick Supplement должен стартовать в `PICKING`, иначе выдача не стартанёт по текущему интерфейсу.

### Степпер / UI шаги
- Степпер **на фронте** должен отличаться (другое количество шагов и подпись), но без изменения бэкенд-логики статусов.
- Бэкенд сохраняет ваши текущие статусы и текущие переходы.

## UI-контракты
### Представление Quick Supplement
Quick Supplement показывается:
- как отдельная заявка (в тех же списках, удобно для логики),
- но выделена цветом/стилем и имеет метку:
  - “Доп. заявка к заявке №{Parent.id}”.

### Метка
Метку достаточно показывать в карточке/строке заявки и в заголовке деталей (чтобы человек мгновенно понял, что это quick доп.).

### Степпер
- Quick Supplement показывает “быстрый” степпер (укороченный, например 3 шага).
- Parent степпер не меняется.

### Запрет действий, “которые бессмысленны”
- Кнопка “Редактировать” для Quick Supplement **не отображается** (в UI это уже делаем по смыслу).
- Бэкенд должен дополнительно запрещать редактирование Quick Supplement (на случай ручных запросов), чтобы ничего не сломать.

## Доп.-услуги: строгий запрет в Quick Supplement
MVP фиксирует правило:
- Quick Supplement создаётся с доп.-услугами выключенными.
- любой UI/форма “быстрой выдачи” не просит вводить delivery/montage/demontage.
- если случайно вызовется путь, где цены доп.-услуг обязательны, система должна вернуть понятную ошибку “Quick Supplement не использует доп.-услуги”.

## Уведомления (Telegram)
### 1) Уведомления текущей системы
Оставляем текущие уведомления “как есть” для стандартных событий (склад начал сборку, склад выдал и т.д.), чтобы не сломать существующее.

### 2) Новое уведомление “быстрая доп.-заявка” для склада
При создании Quick Supplement нужно отправить на склад **новое** уведомление:
- текст должен быть эмоциональный:
  - `🚨 СРОЧНО!! ПРИШЛА БЫСТРАЯ ЗАЯВКА ...`
- далее — содержимое “как при создании новой заявки”:
  - шапка с датами/заказчиком/сотрудником (как в текущих orderHeader),
  - список позиций,
  - без доп.-услуг (в соответствии с договорённостью).

> Примечание: уведомления Greenwich оставляем как текущие по стандартным событиям (`start-picking`, `issue`, `return-declared`, `check-in`), потому что сейчас вы не хотите отдельно усложнять коммуникацию.

## Идемпотентность/надёжность (учёт без потерь данных)
Требование:
- При создании Quick Supplement система должна проверять доступность позиций на дату(ы) Parent так же, как обычная заявка.

Как сделать “без потерь”:
- создание Quick Supplement проходит в транзакции,
- вызывает ту же логику доступности/резервирования, что и обычный POST /api/orders,
- использует те же даты Parent (start/end/readyByDate).

## Список работ для реализации (безопасный MVP)
1. Данные: добавить связь Quick Supplement ↔ Parent (nullable `parentOrderId` и выделение типа).
2. Бэкенд: добавить новый endpoint(ы) “создать quick supplement”:
   - наследует мероприятие по Parent,
   - принимает только строки/позиции из “корзины быстрой выдачи”,
   - запрещает доп.-услуги.
   - Quick Supplement создаётся сразу в “стартовом” статусе (из раздела “Контракт по статусам”), а не в `SUBMITTED`.
3. Бэкенд: запрет редактирования для Quick Supplement:
   - блокировать `PATCH /api/orders/[id]/greenwich-edit` и `PATCH /api/orders/[id]/warehouse-edit` по `parentOrderId`/типу.
   - важно: ваши текущие allowed-status для редактирования включают `APPROVED_BY_GREENWICH` и `PICKING`, поэтому запрет должен быть в бэке, а не только в UI.
4. UI: “быстрый режим” формы/каталога:
   - скрыть ввод заказчика/дат/доп.-услуг,
   - создать quick supplement и показать метку.
5. UI: отрисовать отдельный “быстрый” степпер для quick supplement.
6. Telegram:
   - при создании quick supplement отправить отдельное уведомление “СРОЧНО!! быстрая заявка” складу.
7. Тест-кейсы (минимальный чеклист):
   - Quick создаёт Greenwich: Parent ISSUED → quick APPROVED → start-picking → issue → return/check-in без потерь.
   - Quick создаёт склад: Parent ISSUED → quick PICKING → issue → return/check-in.
   - Нет доп.-услуг: убедиться, что невозможно включить delivery/montage/demontage в quick режиме.
   - Несколько quick к одному Parent: убедиться, что каждый возвращается отдельно.

## Out of scope (после MVP)
- общая приёмка нескольких заявок,
- доп.-услуги в quick,
- интерактивная история “что склеено” в одной форме возврата,
- “склейка” rating/incident компонентов на один родительский возврат.
