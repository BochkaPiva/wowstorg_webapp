# Полный чекап проекта (логика, риски, флоу)

Дата: 2026-03-17 (полный проход). Репозиторий: `wowstorg_webapp`.

## 1. Сильные стороны (уже реализовано хорошо)

- **Критичные операции** с инвентарём и заявками часто обёрнуты в **`prisma.$transaction`** (создание заявки, выдача, приёмка, инциденты, правки строк, quick supplement).
- **Валидация входа** через **Zod** на многих API.
- **Quick supplement**: явная проверка `parentOrderId IS NULL` (только основная заявка), статус `ISSUED`, совпадение Greenwich.
- **Резервирование** (`getReservedQtyByItemId`): пересечение дат `[start,end]` с активными статусами; количество берётся как `issuedQty ?? approvedQty ?? requestedQty`.
- **Уведомления** (Telegram и др.) вызываются с **`void` + `.catch(console.error)`**, чтобы не ломать основной ответ при сбое доставки.
- **Каталог / корзина / клиентские страницы**: где используется цепочка `fetch().then(r => r.json())`, добавлены **`r.json().catch(() => null)`** и **`.catch()`** на цепочку, чтобы сетевой сбой или не-JSON ответ не давали **`unhandledrejection`** (в т.ч. симптом **`[object Event]`** в Next). Покрыто: корзина, каталог, список заявок, очередь/архив склада, аналитика админки, ремонт, правка заявки (каталог позиций в режиме редактирования).

## 2. Риски и ограничения (осознанные)

### 2.1 Гонка при одновременном создании заявок

В **`POST /api/orders`** проверка доступности и создание заказа идут **в одной транзакции**. Реализовано: **`isolationLevel: Serializable`** (PostgreSQL), чтобы параллельные запросы не видели устаревший `reserved`. При откате из‑за конфликта сериализации Prisma отдаёт **`P2034`** → клиенту **`409`** с текстом «Повторите попытку».  
**Дополнительно (если понадобится):** явные блокировки строк `Item`, очередь оформления, retry на клиенте при 409.

### 2.2 Сессия после сбоя сети

В **`providers`** при сбое **`/api/auth/me`** пользователь переводится в **`anonymous`** (как «не залогинен»). Временный обрыв сети может выглядеть как выход.  
**Идея на будущее:** повтор запроса, отдельное состояние «ошибка сети», не сбрасывать сессию с первого же сбоя.

### 2.3 Даты и часовые пояса

В коде смешано **UTC date-only** (поля заявок) и **локальные** подсказки в UI. Для **Омска** часть логики вынесена в отдельные хелперы; при расширении регионов нужна единая политика.

## 3. Исправление по результатам аудита (код)

- **`POST /api/orders/[id]/approve`**: если клиент передаёт строки с неизвестным `orderLineId`, раньше они **молча пропускались** — возможна несогласованная смета. Теперь возвращается **`400`** с указанием неизвестной строки.
- **`POST /api/orders`**: транзакция с **`Serializable`**; ошибки домена (`CUSTOMER_NOT_FOUND`, `ITEM_NOT_FOUND`, `EXCEEDS_AVAILABILITY`) мапятся в **`400`**, а не «сырой» 500.
- **`PATCH .../greenwich-edit`**, **`PATCH .../warehouse-edit`**: проверка **`getReservedQtyByItemId`** перенесена **внутрь** одной **`Serializable`**-транзакции с мутациями строк (устранён TOCTOU: резерв считался до транзакции). Конфликт сериализации → **`409`**.
- **`POST .../quick-supplement/greenwich`** и **`.../warehouse`**: загрузка родителя, резерв и создание дочерней заявки — **одна** **`Serializable`**-транзакция; **`409`** при **`P2034`**.
- **Клиент**: `src/lib/fetchJson.ts` (`readJsonSafe`), усилены **`providers`**, **`AppShell` logout**, **`/orders/[id]`** (load/doAction/save), **`/cart` submit** (`try/catch` на сеть).

## 4. Ручная проверка флоу (чеклист QA)

Рекомендуется пройти в стейджинге/проде по ролям:

| Роль | Флоу |
|------|------|
| Greenwich | Логин → главная → мои заявки (фильтры/поиск) → каталог → корзина → оформление → согласование сметы (если есть) → быстрая доп.-выдача с `ISSUED` |
| Wowstorg | Очередь → смета/сборка/выдача/приёмка → архив → инвентарь/ремонт/утери |
| Admin Wowstorg | Админка → аналитика, пользователи (если доступно) |

Отдельно: **отмена заявки**, **изменения правок**, **возврат на приёмке**, **incident repair/utilize**, **потери found/write-off**.

## 7. Учёт и целостность данных (кратко по коду)

| Область | Как устроено | На что смотреть при изменениях |
|--------|----------------|--------------------------------|
| Резерв на даты | `getReservedQtyByItemId` + пересечение интервалов заявок | Менять формулу количества (`issued` / `approved` / `requested`) согласованно в каталоге и в `POST /api/orders` |
| Создание заявки | Одна **Serializable**-транзакция: резерв + строки + заказ | §2.1; клиент может получить **409** — повтор/сообщение |
| Правка строк (Greenwich / склад) | Резерв и мутации в **одной Serializable**-транзакции | Старый риск «резерв вне tx» снят |
| Quick supplement | Родитель, резерв и `INSERT` в **одной Serializable**-транзакции | Согласовано с `POST /api/orders` |
| Выдача / приёмка / инциденты | Транзакции, списание/возврат по позициям | Любой новый путь изменения `Position` должен проходить через те же инварианты, что существующие API |
| Смета / approve | Неизвестный `orderLineId` → **400** (не молчаливый skip) | Клиент и сервер должны передавать те же id строк |
| Quick supplement | Родитель `ISSUED`, один Greenwich, `parentOrderId IS NULL` у родителя | Не ослаблять проверки при рефакторе |

## 8. Чеклист для новых `"use client"` экранов

- Любой `fetch` → либо `async/await` в `try/catch`, либо `.catch()` на промис; для `res.json()` — **`readJsonSafe`** из `@/lib/fetchJson` или `.catch(() => null)`.
- Не полагаться на то, что API всегда вернёт JSON при 5xx/HTML.

## 9. Полный проход по репозиторию (что просмотрено)

| Зона | Статус |
|------|--------|
| Все `src/app/api/**/route.ts` | Наличие auth/Zod; критичные пути с заявками/резервом усилены (см. §3) |
| `prisma.$transaction` в API | Основные сценарии с резервом дат приведены к **Serializable** + обработка **P2034** где внедрено |
| Клиентские страницы с `fetch` | Ранее и в этом проходе: цепочки `.then(r=>r.json())`, `loadOrder`, корзина, logout |
| `checkout` | Редирект в корзину — отдельной логики нет |
| Остальные транзакции (issue, check-in, approve, инциденты, потери) | Логика списаний внутри tx; отдельный резерв по датам там не дублируется — при сильной конкурренции смотреть **P2034** по аналогии |

**Остаётся на усмотрение:** retry на UI при **409**, улучшение **`/api/auth/me`** при временном обрыве (§2.2), периодический **`npm audit`**.
