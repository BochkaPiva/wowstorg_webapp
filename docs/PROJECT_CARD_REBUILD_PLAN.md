# Пересборка карточки проекта

Дата: 2026-05-22.

## Зачем

Карточка проекта уже покрывает важные сценарии: реквизитные заявки, смету, контакты, файлы, тайминг, журнал и базовые поля проекта. Проблема не в модели, а в скорости ориентации: часть информации дублируется, часть действий спрятана в крупные блоки, а смета визуально выглядит тяжелее, чем должна.

Цель пересборки: чтобы человек открыл проект и за 10-20 секунд понял, что происходит, что блокирует работу, чья очередь действовать, какие заявки активны, какие задачи по проекту висят и что нужно сделать дальше.

## Текущее состояние

### Что уже хорошо

- `Project` уже хранит нужные поля шапки: `title`, `status`, `ball`, `eventStartDate`, `eventEndDate`, `eventDateNote`, `eventDateConfirmed`, `openBlockers`, `internalSummary`, `archivedAt`.
- `Order.projectId` связывает проект с заявками; блок "Заявки реквизита" уже показывает заявки и использует `OrderStatusStepper`.
- `WorkTask.projectId` и `WorkTask.orderId` уже есть в Prisma, поэтому проектный YouGile можно строить как представление тех же задач, а не как новую сущность.
- API задач уже принимает `projectId` и `orderId` при создании/редактировании задачи.
- Архив проекта уже отделен от бизнес-статуса через `archivedAt`; архивные проекты становятся read-only.

### Что мешает

- Блок "Карточка проекта" дублирует шапку: название, статус, мяч, даты и заметки частично повторяются.
- Статус и мяч меняются через один большой режим редактирования с длинным списком статусов. Это технически работает, но плохо помогает принять следующее действие.
- "Рабочие заметки" полезны, но занимают слишком много места при пустых или коротких значениях.
- В проекте нет встроенной доски задач, хотя модель задач уже поддерживает связь с проектом.
- Основная доска задач сейчас получает все задачи доски без серверного фильтра по проекту/активности проекта.
- Смета сохраняет нужные поля, но визуально перегружена карточками, плашками и повторяющимися суммами.
- Нижние финансовые итоги показывают цифры, но не группируют смысл: клиентская сумма, внутренние затраты, маржа.

## Принципы пересборки

1. Не менять модель данных без необходимости. В первую волну достаточно UI и небольших API-фильтров.
2. Не удалять полезные ориентиры. Степпер заявки оставить: он быстро объясняет состояние реквизита.
3. Проектный YouGile должен быть теми же задачами, что и основной YouGile, только с фильтром `projectId`.
4. Закрытые/отмененные/архивные проекты не должны шуметь в основной доске, но их задачи не нужно удалять или отвязывать.
5. Смета должна остаться функционально полной: все текущие поля сохраняются, меняется упаковка и визуальная иерархия.

## Новая структура страницы

### 1. Шапка проекта как главный центр управления

Шапка заменяет отдельный блок "Карточка проекта".

В шапке остаются:

- название проекта, редактируемое inline;
- заказчик;
- ответственный;
- даты мероприятия и флаг подтверждения дат;
- статус проекта;
- мяч;
- количество/состояние заявок;
- быстрые действия.

Редактирование:

- название редактируется по клику на текст или иконку;
- даты редактируются компактным поповером;
- статус и мяч редактируются отдельными маленькими контролами, а не одним общим длинным режимом.

### 2. Новый контроль статуса

Вместо большого выпадающего списка:

- основная кнопка: рекомендуемый следующий статус;
- рядом меню "Другой статус";
- статусы сгруппированы по этапам:
  - продажа и бриф;
  - КП и договор;
  - подготовка;
  - проведение;
  - закрытие;
  - пауза/отмена.

Пример логики:

| Текущий статус | Главная кнопка |
|---|---|
| `LEAD` | "В сбор брифа" |
| `BRIEFING` | "Во внутреннюю подготовку" |
| `INTERNAL_PREP` | "КП отправлено" |
| `PROPOSAL_SENT` | "Правки КП" или "Договор готовим" |
| `CONTRACT_SIGNED` | "Предпродакшн" |
| `READY_TO_RUN` | "Проведение" |
| `LIVE` | "Закрытие" |
| `WRAP_UP` | "Завершить" |

Смена статуса может предлагать мяч по умолчанию, но не должна жестко перетирать ручной выбор без подтверждения.

### 3. Новый контроль мяча

Мяч лучше сделать не выпадающим списком, а сегментированным контролом:

- `CLIENT` - клиент;
- `WOWSTORG` - мы;
- `VENDOR` - подрядчик;
- `VENUE` - площадка;
- `NONE` - нет активного ожидания.

Это уже есть в enum `ProjectBall`, миграция не нужна.

### 4. Компактные рабочие заметки

Оставить два смысла:

- "Блокеры" (`openBlockers`) - что мешает прямо сейчас.
- "Резюме" (`internalSummary`) - контекст для коллеги.

Новая упаковка:

- если оба поля пустые, показывать одну компактную строку "Добавить блокер / резюме";
- если заполнены, показывать две компактные заметки рядом или одну под другой на мобильном;
- редактирование через inline-поповер или раскрытие, без большого пустого блока.

### 5. Контакты

Порядок блока оставить. Улучшить только визуальную плотность:

- карточки контактов компактнее;
- последнюю запись переговоров показывать прямо в карточке контакта;
- действие "Добавить запись" оставить на карточке контакта, но сделать менее тяжелым визуально.

Это можно делать после шапки и задач, потому что контакты сейчас не главный источник боли.

### 6. Заявки реквизита

Степпер оставить. Его роль: быстро показать, где находится заявка.

Улучшения:

- список заявок можно сделать чуть плотнее;
- детали заявки раскрывать только по необходимости;
- iframe заявки оставить как управленческий режим, но он не должен съедать внимание в свернутом состоянии;
- кнопка "Каталог -> реквизит" остается главным входом.

### 7. Проектный YouGile

Добавить отдельный блок "Задачи проекта" между заметками/контактами и заявками либо как первую вкладку рабочей зоны. Рекомендация: отдельный блок до заявок, потому что задачи отвечают на вопрос "что делать", а заявки - на вопрос "что с реквизитом".

Поведение:

- внутри проекта показываются задачи, где `WorkTask.projectId = текущий project.id`;
- создание задачи из проекта автоматически проставляет `projectId`;
- если в основном YouGile задаче выбрать этот проект, она появляется в проектном блоке;
- перенос между колонками работает на тех же `WorkTaskColumn`, поэтому статус задачи синхронизирован без отдельной логики;
- подзадачи, inline-название и inline-описание использовать из текущего `/tasks`, чтобы не плодить второй UX.

Основной YouGile:

- по умолчанию показывает задачи без привязки к архивным/завершенным/отмененным проектам;
- задачи закрытых проектов не удаляются, а скрываются серверным фильтром;
- нужен явный фильтр "Показать задачи закрытых проектов" для истории и поиска.

Технически лучше не создавать отдельные проектные доски. Доска и колонки остаются общими, а проектная карточка показывает срез задач по `projectId`. Это дает настоящую двустороннюю синхронизацию без фоновой синхронизации.

### 8. Рабочая зона

Оставить вкладки:

- Смета;
- Тайминг;
- Файлы;
- Журнал.

Добавлять "Задачи" во вкладки не обязательно, если задачи вынесены отдельным видимым блоком. Если страница станет слишком длинной, можно сделать задачи первой вкладкой рабочей зоны, но тогда хуже видимость ежедневных дел.

## Смета

### Цель редизайна

Сохранить все текущие поля и бизнес-логику, но сделать смету похожей не на набор тяжелых карточек, а на рабочую таблицу с понятными группами.

### Разделы

Секции оставить:

- `REQUISITE` - реквизит из заявки;
- `DRAFT_REQUISITE` - demo-реквизит;
- `LOCAL` - локальные строки;
- `CONTRACTOR` - подрядчики.

Визуально:

- заголовок секции должен показывать тип, название, связанную заявку/дату и сумму;
- тело секции раскрыто только у активной секции;
- управление секцией находится справа в заголовке, а не размазано по блоку.

### Строки сметы

На desktop сделать более табличную сетку:

- позиция;
- описание;
- ед.;
- кол-во;
- дней;
- цена/ед.;
- сумма;
- внутренняя цена/затраты;
- статус оплаты/комментарии для подрядчиков.

На mobile оставить карточный вид.

Важно:

- редактируемые поля должны выглядеть как поля;
- рассчитанные поля должны выглядеть как read-only значения;
- сумма строки не должна конкурировать визуально с итогами секции.

### Нижние итоги

Пересобрать в 3 смысловые группы:

1. Клиент:
   - сумма клиенту;
   - условный налог 6%;
   - комиссия 15%;
   - итог клиенту.
2. Внутреннее:
   - себестоимость;
   - валовая маржа.
3. Маржа:
   - маржа после налога;
   - маржа после налога, %.

Визуально это один компактный summary-блок, а не много равноправных карточек.

## API и данные

### Что можно сделать без миграции

- Удалить/заменить дублирующий UI-блок "Карточка проекта".
- Перестроить шапку и заметки.
- Добавить проектный YouGile как фильтр существующих задач.
- Добавить API-фильтры к текущим endpoint задач:
  - `GET /api/tasks/boards/[id]?projectId=...`;
  - `GET /api/tasks/boards/[id]?includeClosedProjectTasks=1`;
  - опционально `scope=main|project`.
- При создании задачи из проекта использовать существующее поле `projectId`.
- В основной доске скрывать задачи закрытых/архивных проектов через `where` по связанной `project`.

### Где нужна осторожность

- `GET /api/tasks/boards/[id]` сейчас загружает задачи без фильтра по проекту и без проверки статуса проекта. Изменение фильтра затронет основной YouGile.
- `PATCH /api/tasks/tasks/[id]` уже умеет менять `projectId`, `orderId`, `columnId`, `completed`; нужно не сломать текущую оптимистичную перетаскивалку.
- `Project.archivedAt` запрещает редактирование проекта, но задачи проекта технически остаются редактируемыми через общий YouGile. Нужно решить: в первой итерации скрываем их из основной доски и делаем проект read-only; отдельно можно запретить редактирование задач архивного проекта сервером.
- Закрытие проекта сейчас возможно только при терминальном статусе и закрытых/отмененных заявках. Это правило сохраняем.

## Компонентный план

### Новый или выделяемый UI

- `ProjectHeaderPanel` - шапка, inline title/date/actions/status/ball.
- `ProjectStatusControl` - рекомендованный следующий статус + меню всех статусов.
- `ProjectBallControl` - сегментированный контроль мяча.
- `ProjectNotesStrip` - компактные блокеры и резюме.
- `ProjectTasksBoard` - проектный срез YouGile.
- `TaskBoardView` или общий компонент из `/tasks` - общий визуальный слой доски, чтобы основной и проектный YouGile не расходились.
- `EstimateFinancialSummary` - новый нижний блок итогов.
- `EstimateSectionHeader` и `EstimateLineGrid` - более табличная упаковка сметы.

### Что лучше вынести из `src/app/tasks/page.tsx`

Текущая страница задач содержит много полезной логики внутри одного файла. Для переиспользования в проекте стоит вынести:

- типы задач/колонок;
- `RoundCheckbox`;
- `TaskCard`;
- дерево подзадач;
- функции чтения/патча задач;
- board view с колонками и drag-and-drop.

Сначала можно вынести минимальный общий компонент без большого рефакторинга: `TaskBoardView` принимает `board`, callbacks и режим `projectId`.

## Фазы реализации

### Фаза 1. Шапка и удаление дублей

- Убрать отдельный блок "Карточка проекта".
- Перенести редактирование названия, дат, статуса и мяча в шапку.
- Сделать `ProjectStatusControl` и `ProjectBallControl`.
- Сжать "Рабочие заметки".

Приемка:

- все текущие поля проекта редактируются;
- статус и мяч не требуют открытия большого блока;
- нет дублирования "статус/мяч/даты" ниже шапки;
- архивный проект остается read-only.

### Фаза 2. Проектный YouGile

- Вынести общий компонент доски из `/tasks`.
- Добавить серверный фильтр `projectId` в `GET /api/tasks/boards/[id]`.
- Добавить скрытие задач закрытых/архивных проектов в основной доске по умолчанию.
- Добавить блок задач на страницу проекта.
- Создание задачи из проекта автоматически проставляет `projectId`.

Приемка:

- задача, созданная в проекте, видна в основном YouGile;
- задача, привязанная к проекту в основном YouGile, видна в проекте;
- перенос по колонкам синхронен в обоих местах;
- задачи закрытого/архивного проекта не шумят в основной доске;
- задачи не удаляются при закрытии проекта.

### Фаза 3. Смета и финансы

- Пересобрать визуал строк сметы без изменения полей.
- Сделать более понятные заголовки секций.
- Заменить нижние карточки итогов на 3 смысловые группы.
- Проверить XLSX/PDF контракты не меняются.

Приемка:

- все старые поля доступны;
- суммы совпадают с текущей логикой;
- реквизитные блоки по заявкам остаются 1:1;
- desktop выглядит как рабочая таблица, mobile - как компактные карточки.

### Фаза 4. Полировка

- Проверить адаптивность.
- Проверить сценарии archive/read-only.
- Прогнать `npm run build`.
- При изменении API - `npm run brain:inventory`.
- Обновить feature-документацию, если по ходу появятся новые endpoint/env.

## Открытые вопросы перед кодом

1. Скрывать ли задачи проекта из основной доски уже при `status in (COMPLETED, CANCELLED)` или только при `archivedAt != null`? Рекомендация: скрывать при любом из этих условий, но добавить фильтр "Показать закрытые".
2. Разрешать ли редактировать задачи архивного проекта из основного YouGile, если включен фильтр показа? Рекомендация: нет, только просмотр.
3. Где поставить блок "Задачи проекта": отдельным блоком до заявок или первой вкладкой рабочей зоны? Рекомендация: отдельным блоком до заявок.
4. Автоматически менять мяч при смене статуса или только предлагать? Рекомендация: предлагать, но явно показывать изменение.

## Риски

- Самый высокий риск - случайно разорвать текущий YouGile при выносе общего компонента. Поэтому доску лучше выделять маленькими шагами.
- Второй риск - изменить сметную математику вместе с визуалом. В фазе 3 нельзя менять формулы без отдельного решения.
- Третий риск - сделать статус слишком "умным" и неожиданно менять мяч. Контроль должен помогать, а не принимать решение молча.
