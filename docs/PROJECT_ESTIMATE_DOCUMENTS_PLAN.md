# Несколько смет в одном проекте

Документ фиксирует целевую модель пересборки блока сметы проекта. Текущая реализация использует `ProjectEstimateVersion` как версии одной сметы. Целевая модель проще: в проекте есть несколько самостоятельных смет, без бизнес-версионирования.

## 1. Цель

Сделать сметы проекта ближе к реальной работе:

- в одном проекте может быть несколько отдельных смет;
- каждую смету можно назвать вручную;
- заявки реквизита добавляются в смету только вручную;
- одна заявка проекта может быть добавлена только в одну смету;
- финансы проекта считаются как сумма всех учитываемых смет;
- старые редакции смет не храним как версии в базе, а при необходимости сохраняем XLSX-файлами.

## 2. Что убираем из целевой логики

### Версии смет

Бизнес-версионирование не нужно. Если смету поправили, она просто изменилась. Если нужно сохранить старую редакцию, пользователь выгружает XLSX и хранит файл в проекте.

Текущее понятие `ProjectEstimateVersion` можно использовать как технический переходный слой, но в UI и доменной логике пользователь должен видеть не версии, а сметы:

- "Смета проекта";
- "Доп. работы";
- "Техническая часть";
- "Доп. соглашение N1";
- любое другое ручное название.

### Статусы смет

Статусы смет в MVP не добавляем. Сейчас процесс работает без них, а дополнительные состояния создадут лишние вопросы: когда смета "согласована", когда "отправлена", кто и где это подтверждает.

Для текущей задачи достаточно:

- смета активна и участвует в финансах;
- смета не участвует в финансах;
- смета удалена или скрыта, если это будет нужно отдельной итерацией.

## 3. Термины

### Проект

Контейнер мероприятия, задач, заявок, файлов, тайминга и финансов.

### Смета проекта

Самостоятельный финансовый документ внутри проекта. Все сметы равноправны. Название задаётся пользователем.

Примеры:

- "Основная смета";
- "Доп. соглашение по реквизиту";
- "Техническая часть";
- "Доп. работы по монтажу";
- "Вариант для клиента".

### Заявка реквизита

Обычная `Order`, связанная с проектом через `projectId`. В смету она попадает не автоматически, а через ручное действие "Добавить заявки".

### Ручной раздел

Раздел внутри сметы, который не связан с заявкой: подрядчики, услуги, техника, организационные расходы и другие позиции.

## 4. Главные правила

### 4.1. В проекте может быть много смет

Пользователь может создать любое количество смет внутри проекта. Смета создаётся пустой, без автоматического добавления заявок.

Исключение допустимо только для миграции: для существующих проектов текущая смета переносится в одну смету с названием "Смета проекта".

### 4.2. Название сметы свободное

Не вводим жёсткие типы "основная", "допник", "техническая". Это пользовательские сценарии, а не обязательная структура данных.

Минимальные поля сметы:

- название;
- порядок отображения;
- флаг "учитывать в финансах проекта";
- даты создания/обновления;
- автор создания.

### 4.3. Заявки добавляются вручную

Внутри каждой сметы есть блок "Заявки реквизита" и кнопка "Добавить заявки".

При создании новой сметы заявки туда не попадают автоматически.

### 4.4. Одна заявка только в одной смете проекта

Это ключевое правило против двойного учета.

Одна `Order` с `projectId` может быть привязана только к одной смете этого проекта. Если заявка уже добавлена в одну смету, добавить её во вторую нельзя.

Если заявку нужно перенести:

1. убрать заявку из текущей сметы;
2. добавить её в другую смету.

Позже можно сделать отдельное действие "Перенести в другую смету", но в первой реализации это не обязательно.

### 4.5. Финансы проекта не считают заявку дважды

Так как заявка может быть только в одной смете, проектные финансы считаются проще и безопаснее:

```text
Финансы проекта =
  смета 1
+ смета 2
+ смета 3
```

Двойной учет заявок становится невозможным на уровне данных и UI.

### 4.6. Ручные разделы принадлежат конкретной смете

Если ручная строка создана в смете "Доп. работы", она влияет только на эту смету и на общий итог проекта, если эта смета учитывается в финансах.

## 5. Целевая модель данных

### 5.1. Новый доменный уровень

Нужна сущность между `Project` и разделами сметы:

```prisma
model ProjectEstimateDocument {
  id                      String   @id @default(cuid())
  projectId               String
  title                   String
  sortOrder               Int
  includeInProjectTotals  Boolean  @default(true)
  createdById             String
  createdAt               DateTime @default(now())
  updatedAt               DateTime @updatedAt

  project                 Project  @relation(fields: [projectId], references: [id], onDelete: Cascade)
  createdBy               User     @relation(fields: [createdById], references: [id], onDelete: Restrict)
  sections                ProjectEstimateSection[]

  @@index([projectId, sortOrder])
  @@index([projectId, includeInProjectTotals])
}
```

Название `ProjectEstimateDocument` рабочее. В UI это просто "смета".

### 5.2. Разделы сметы

Текущий `ProjectEstimateSection` должен быть привязан к смете, а не к версии:

```prisma
model ProjectEstimateSection {
  id              String @id @default(cuid())
  estimateId      String
  estimate        ProjectEstimateDocument @relation(fields: [estimateId], references: [id], onDelete: Cascade)
  sortOrder       Int
  title           String
  kind            ProjectEstimateSectionKind
  linkedOrderId   String?
  lineLocalExtras Json?
  lines           ProjectEstimateLine[]

  @@index([estimateId, sortOrder])
  @@index([linkedOrderId])
}
```

Для переходного периода можно сначала добавить `estimateId` nullable, перенести данные, затем сделать обязательным.

### 5.3. Запрет одной заявки в нескольких сметах

Нужно гарантировать уникальность `linkedOrderId` в рамках проектных смет.

Так как `ProjectEstimateSection` напрямую не хранит `projectId`, самый простой надёжный вариант:

- не добавлять заявку в смету, если уже есть секция с этим `linkedOrderId` в любой смете проекта;
- дополнительно рассмотреть техническую таблицу связей, если понадобится строгий уникальный индекс.

Более строгая модель:

```prisma
model ProjectEstimateOrderLink {
  id         String @id @default(cuid())
  projectId  String
  estimateId String
  orderId    String
  createdAt  DateTime @default(now())

  @@unique([projectId, orderId])
  @@index([estimateId])
}
```

Но в первой реализации можно обойтись без отдельной таблицы, если секция `REQUISITE` остаётся единственным способом привязать заявку к смете.

### 5.4. Что делать с `ProjectEstimateVersion`

Текущий `ProjectEstimateVersion` сейчас используется много где: API, UI, draft-order, аналитика, экспорт. Безопаснее не удалять его одним ударом.

Рекомендуемый путь:

1. Добавить `ProjectEstimateDocument`.
2. Создать по одной смете для каждого проекта, где есть текущие версии.
3. Привязать актуальную версию/её секции к новой смете.
4. В UI перестать показывать версии.
5. После стабилизации удалить или законсервировать старую модель версий.

Если нужно ускорить реализацию, можно временно переиспользовать `ProjectEstimateVersion` как "смету":

- `versionNumber` оставить техническим;
- `note` или новое поле `title` использовать как название сметы;
- `isPrimary` заменить или игнорировать;
- запретить создание "версий" в UI, показывая их как отдельные сметы.

Но это компромисс. Чистее всё-таки выделить новую сущность сметы.

## 6. UI: блок смет проекта

### 6.1. Общий вид

Блок должен быть понятным без чтения длинных пояснений:

```text
Сметы проекта                                      + Новая смета

[ Смета проекта       42 612 ₽   прибыль 39 456 ₽   в финансах ]
[ Доп. работы         18 000 ₽   прибыль 11 200 ₽   в финансах ]
[ Вариант клиента     60 000 ₽   прибыль 0 ₽        не учитывается ]
```

Клик по смете открывает её редактор ниже.

### 6.2. Создание сметы

Кнопка: "Новая смета".

Модальное окно:

- название сметы;
- переключатель "Учитывать в финансах проекта";
- кнопка "Создать".

По умолчанию:

- название пустое или "Новая смета";
- "Учитывать в финансах проекта" включено.

Смета создаётся пустой.

### 6.3. Редактор выбранной сметы

В шапке редактора:

- название, редактируемое inline;
- итог клиенту;
- себестоимость;
- прибыль;
- маржа;
- переключатель "Учитывать в финансах";
- кнопки XLSX внутренний / XLSX клиент;
- меню действий.

Ниже:

1. "Заявки реквизита";
2. ручные разделы;
3. итоги выбранной сметы.

### 6.4. Добавление заявок

Кнопка "Добавить заявки" открывает модальное окно.

В модалке список заявок проекта:

```text
[ ] Форум Атриум · 21.05-22.05 · 42 612 ₽
[ ] Доп. реквизит · 22.05 · 18 000 ₽
[-] Уже в смете "Техническая часть" · 12 000 ₽
```

Правила:

- свободные заявки можно выбрать чекбоксами;
- заявки, уже добавленные в другую смету, disabled;
- рядом показывается, в какой смете они уже находятся;
- кнопка "Добавить выбранные".

Если свободных заявок нет, показываем спокойное пустое состояние:

```text
Все заявки проекта уже распределены по сметам.
```

### 6.5. Заявка внутри сметы

Заявка отображается компактным блоком:

```text
Форум Атриум                       42 612 ₽
21.05.2026 — 22.05.2026            4 позиции
[Открыть заявку] [Убрать из сметы]
```

"Убрать из сметы" не удаляет заявку и не отменяет её. Это только отвязка от сметы.

Если заявка отменена, блок должен быть приглушён и явно показывать, что она не участвует в деньгах.

### 6.6. Ручные разделы

Ручные разделы остаются такими же по смыслу:

- универсальный раздел;
- подрядчики;
- будущие типы, если понадобятся.

Раздел создаётся внутри выбранной сметы и не виден в других сметах.

## 7. Финансовые правила

### 7.1. Итог отдельной сметы

Итог сметы считается из:

- заявок реквизита, добавленных в эту смету;
- ручных разделов этой сметы.

Показываем:

- сумма клиенту;
- комиссия, если включена в формулах;
- налог;
- себестоимость;
- расходы на наличку, если применимо;
- прибыль;
- маржа.

Формулы должны использовать текущие функции проекта, а не новую параллельную математику.

### 7.2. Итог проекта

Итог проекта считается как сумма всех смет, где `includeInProjectTotals = true`.

Не учитываются:

- сметы с выключенным флагом "Учитывать в финансах";
- удалённые сметы, если появится мягкое удаление;
- отменённые заявки внутри смет;
- отменённые проекты в общей аналитике, как сейчас.

### 7.3. Заявки в аналитике

Заявка, привязанная к проекту, не должна считаться как самостоятельная заявка в финансовой аналитике. Её деньги входят в проект через смету, к которой она добавлена.

Если заявка проекта ещё не добавлена ни в одну смету:

- в карточке проекта нужен сигнал "Есть заявка вне сметы";
- в финансах проекта она не должна теряться без предупреждения;
- рекомендуемое поведение для аналитики: не включать её в проектные финансы, но показывать в блоке контроля как нераспределённую заявку проекта.

Это важная защита от ситуации, когда деньги есть в заявке, но не попали в смету проекта.

### 7.4. Сметы без заявок

Смета может состоять только из ручных разделов. Это нормальный сценарий для услуг, подрядчиков, координации, технической части и доп. работ.

## 8. API: целевой контракт

### 8.1. Список смет проекта

```http
GET /api/projects/:id/estimates
```

Ответ:

```json
{
  "estimates": [
    {
      "id": "est_1",
      "title": "Смета проекта",
      "sortOrder": 0,
      "includeInProjectTotals": true,
      "totals": {
        "clientTotal": 42612,
        "internalTotal": 4000,
        "marginAfterTax": 39456,
        "marginAfterTaxPct": 85
      }
    }
  ],
  "projectTotals": {
    "clientTotal": 42612,
    "internalTotal": 4000,
    "marginAfterTax": 39456,
    "marginAfterTaxPct": 85
  }
}
```

### 8.2. Создать смету

```http
POST /api/projects/:id/estimates
```

Тело:

```json
{
  "title": "Доп. работы",
  "includeInProjectTotals": true
}
```

### 8.3. Обновить смету

```http
PATCH /api/projects/:id/estimates/:estimateId
```

Тело:

```json
{
  "title": "Техническая часть",
  "includeInProjectTotals": false
}
```

### 8.4. Добавить заявки в смету

```http
POST /api/projects/:id/estimates/:estimateId/orders
```

Тело:

```json
{
  "orderIds": ["order_1", "order_2"]
}
```

Проверки:

- все заявки принадлежат этому проекту;
- ни одна заявка уже не добавлена в другую смету этого проекта;
- заявка не отменена, либо отменённые можно добавлять только если есть осознанная причина. Рекомендуемое MVP: отменённые не добавлять.

Если заявка уже занята:

```json
{
  "error": "ORDER_ALREADY_IN_ESTIMATE",
  "message": "Заявка уже добавлена в смету \"Техническая часть\"."
}
```

### 8.5. Убрать заявку из сметы

```http
DELETE /api/projects/:id/estimates/:estimateId/orders/:orderId
```

Удаляет только связь/секцию реквизита из сметы. Саму заявку не трогает.

### 8.6. XLSX

```http
GET /api/projects/:id/estimates/:estimateId/xlsx?variant=client
GET /api/projects/:id/estimates/:estimateId/xlsx?variant=internal
```

Экспорт всегда относится к одной конкретной смете, не ко всему проекту.

Отдельный общий проектный финансовый отчёт можно сделать позже.

## 9. Миграция существующих данных

### 9.1. Для каждого проекта

Если у проекта есть текущая смета:

1. создать `ProjectEstimateDocument` с названием "Смета проекта";
2. перенести в неё актуальные разделы текущей основной версии;
3. сохранить текущие строки, локальные extras и привязки к заявкам;
4. выставить `includeInProjectTotals = true`.

### 9.2. Если у проекта несколько версий

Так как версии больше не являются бизнес-сущностью, нужно выбрать одну актуальную:

1. если есть `isPrimary = true`, переносим её;
2. иначе переносим последнюю по `versionNumber` или `createdAt`;
3. остальные версии не переносим в активную модель.

Если нужно сохранить старые версии, перед миграцией можно выгрузить их XLSX вручную. Автоматически тащить все версии как отдельные сметы нельзя: это создаст риск двойного учёта.

### 9.3. Draft/demo заявки

`ProjectDraftOrder.estimateVersionId` сейчас указывает на версию. В новой модели нужно заменить привязку на `estimateId`.

Логика:

- demo-заявка принадлежит выбранной смете;
- при материализации в реальную заявку она остаётся в этой же смете;
- если пользователь создаёт demo из проекта, UI должен явно понимать, в какую смету она попадёт.

## 10. Обновление аналитики

Затронутые места:

- список проектов;
- карточка проекта;
- админская финансовая аналитика;
- XLSX финансовый отчёт;
- дашборд, если он показывает финансы проектов.

Новая логика:

- проектная выручка = сумма учитываемых смет проекта;
- проектная прибыль = сумма прибыли учитываемых смет проекта;
- заявка с `projectId` не участвует в самостоятельных заявках;
- заявка проекта, не добавленная ни в одну смету, должна попасть в сигнал контроля, а не молча исчезнуть.

## 11. Риски

### Риск 1. Потерять деньги заявки, если она не добавлена в смету

Защита:

- блок "Нераспределённые заявки проекта";
- предупреждение в карточке проекта;
- предупреждение в админской аналитике.

### Риск 2. Случайно удалить связь заявки со сметой

Защита:

- действие "Убрать из сметы" должно быть явно подписано;
- для заявок с деньгами можно показывать подтверждение;
- сама заявка не удаляется.

### Риск 3. Сломать старые проекты

Защита:

- миграция создаёт одну смету на проект;
- текущий UI можно сначала адаптировать к одной смете;
- только после этого включать создание нескольких смет.

### Риск 4. Перепутать "не учитывать в финансах" и "удалить"

Защита:

- в UI флаг должен звучать как "Учитывать в итогах проекта";
- рядом короткая легенда: "Выключи, если это черновик или вариант, который не должен попадать в прибыль проекта."

## 12. План реализации

### Фаза 1. Подготовка модели

- Добавить `ProjectEstimateDocument`.
- Добавить связь секций со сметой.
- Подготовить миграцию существующих данных.
- Оставить старые API совместимыми на время перехода.

### Фаза 2. API смет

- Список смет проекта.
- Создание/переименование сметы.
- Переключатель "Учитывать в итогах проекта".
- Добавление заявок в смету.
- Удаление заявки из сметы.
- Запрет добавления заявки в две сметы.

### Фаза 3. UI блока смет

- Верхний список смет.
- Модалка создания сметы.
- Модалка выбора заявок чекбоксами.
- Редактор выбранной сметы.
- Сигналы по нераспределённым заявкам.

### Фаза 4. Финансы

- Пересчитать карточку проекта по сумме смет.
- Обновить список проектов.
- Обновить админскую аналитику.
- Обновить финансовый XLSX-отчёт.
- Проверить, что заявки в проектах не считаются отдельно.

### Фаза 5. Уборка версий

## 13. Приёмка

- [ ] В проекте можно создать несколько смет с ручными названиями.
- [ ] Новая смета создаётся пустой.
- [ ] В смету можно добавить одну или несколько заявок через модалку с чекбоксами.
- [ ] Заявку, уже добавленную в смету, нельзя добавить в другую смету проекта.
- [ ] В модалке видно, в какой смете уже находится занятая заявка.
- [ ] Заявку можно убрать из сметы, не удаляя саму заявку.
- [ ] Ручные разделы принадлежат выбранной смете.
- [ ] Итог сметы считается только по её заявкам и ручным разделам.
- [ ] Итог проекта считается по всем сметам с включённым флагом "Учитывать в итогах проекта".
- [ ] Заявки с `projectId` не считаются отдельно в финансовой аналитике заявок.
- [ ] Если заявка проекта не добавлена ни в одну смету, система показывает сигнал.
- [ ] XLSX выгружается по конкретной смете.
- [ ] Старые проекты после миграции получают одну смету "Смета проекта" с текущими данными.
