# План: «Утро / вечер» для дат аренды (оплачиваемые дни мероприятия)

Документ описывает **узлы системы**, которые затронет фича, **целевую архитектуру** (единая формула оплаты на сервере), **фазы внедрения**, **риски** и **открытые решения** до начала кода.

Референс по UI: переключатель из Uiverse (andrew-demchenk0), разметка и стили переданы владельцем продукта — использовать как визуальный паттерн; в реализации завернуть в доступный компонент (см. §6).

---

## 1. Цель продукта

Пользователь явно указывает **часть календарного дня** для даты **начала** и **окончания** мероприятия (условно «утро» / «вечер»). Из пары дат + двух флагов вычисляется **количество оплачиваемых дней аренды** (`billableRentalDays`), на которое умножается аренда позиций (как сейчас — через существующую модель скидок, налога и `payMultiplier`).

Инвариант: **сумма в каталоге/корзине = сумма на сервере** для тех же входных данных.

## 3. Модель данных (предложение)

Добавить на `Order` два поля (имена обсуждаемы):

- `rentalStartPartOfDay` — enum: `MORNING` | `EVENING` (или `DAY_START` / `DAY_END` — избегать путаницы с «концом суток»).
- `rentalEndPartOfDay` — то же.

**Дефолт для миграции:** зафиксировать в ADR (например оба `MORNING`), чтобы **старые заявки** после выката вели себя предсказуемо и при необходимости совпадали с прежней коммерческой интерпретацией дат на проде (после отката — уточнить у владельца).

Nullable не рекомендуется: всегда два значения; миграция заполняет дефолтом.

---

## 4. Формула `billableRentalDays` (блокер: матрица)

Сейчас от продукта заданы **примеры**, а не полная спецификация:

- начало **утро**, конец **вечер** → **2** дня (в типичном смысле «два календарных дня мероприятия»);
- **утро** — **утро** → **1** день;
- **вечер** — **утро** (на следующий календарный день) → **1** день.

**Обязательно до кодирования:** таблица **4×4** для всех сочетаний `(startPart, endPart)` и **всех отношений дат** `startDate` vs `endDate`:

- `startDate === endDate` (один календарный день): утро→вечер, утро→утро, вечер→вечер, вечер→утро — сколько платим?
- `endDate === startDate + 1`: все четыре комбинации;
- `endDate > startDate + 1`: линейное правило (например базовые дни по календарю + поправка на края) или поштучная таблица.

Рекомендация инженерной записи правила после утверждения продукта:

```text
billableRentalDays = clampMin1( … )
```

реализация в **одном** модуле, например расширение `src/lib/rental-days.ts` + использование из `order-pricing.ts` (без дублирования на клиенте кроме тонкой копии той же функции из `lib`).

Если понадобятся **дробные дни** (0.5) — заранее решить округление суммы по строке и общий минимум; иначе оставаться в **целых днях** с явной таблицей.

## 5. API и транзакции

| Маршрут / слой | Изменение |
|----------------|-----------|
| `POST /api/orders` | Zod: `rentalStartPartOfDay`, `rentalEndPartOfDay`; передать в `createOrderInTransaction`. |
| `create-order.ts` | Валидация согласованности с датами; запись полей в `Order`; `calcOrderPricing` с новыми аргументами. |
| `GET /api/orders/[id]` | Вернуть оба поля (строки enum). |
| `PATCH .../greenwich-edit`, `.../warehouse-edit` | Разрешить правку в тех же статусах, где сейчас можно менять даты; валидация + snapshots при необходимости. |
| Quick supplement | Наследовать от родительской заявки или переопределять — решение в продукте; задокументировать. |

Клиентские превью (каталог, корзина, карточка заказа) вызывают **ту же** функцию из `@/lib/...`, что сервер будет использовать после парсинга дат — **никаких** «особых только в каталоге» формул.

## 6. UI (ваш переключатель)

Исходник: `<label class="switch">` + `.sun`/`.moon` + скрытый `input.checkbox`.

Рекомендации при внедрении:

1. **Доступность:** `role="switch"`, `aria-checked`, подпись «Начало: утро или вечер» / «Окончание: …». Чекбокс остаётся реальным `input`, связь через `aria-labelledby` или видимый текст.
2. **Смысл checked:** явно закрепить в коде комментарием и в карточке фичи: например `checked === EVENING`, `unchecked === MORNING` (или наоборот — как решит UX).
3. **Изоляция стилей:** вынести CSS в модуль каталога/общий `*.css`/Tailwind `@layer`, класс `.switch` переименовать во что-то вроде `.wowstorg-partofday-switch`, чтобы не пересечься с глобалами.
4. **Анимации:** текущие `rotate`/`tilt` можно оставить; при желании включать только `@media (prefers-reduced-motion: no-preference)`.

Размещение: под полями **Дата начала** и **Дата окончания** по одному переключателю; в корзине — те же элементы синхронно с `catalogDates`/расширенным storage.

При отправке сметы / согласовании полезно в snapshot (или в человекочитаемую строку периода) включать:

## 9. Фазы внедрения (рекомендуемый порядок)

### Фаза 0 — продукт (блокер)

- Утвердить полную матрицу `billableRentalDays`.
- Зафиксировать дефолты для legacy-строк БД и поведение quick supplement.

### Фаза 1 — доменное ядро

- Новый модуль или расширение `rental-days`: `billableRentalDays(...)`.
- `calcOrderPricing`: новый параметр `{ startPartOfDay, endPartOfDay }`, внутри заменить использование голого `daysBetween` для аренды на `billableRentalDays`.
- Юнит-тесты или минимальный `node`-скрипт с таблицей кейсов от продукта (желательно).

### Фаза 2 — БД и создание заказа

- Prisma migration + `create-order` + `POST /api/orders`.
- `GET /api/orders/[id]` отдаёт поля.

### Фаза 3 — клиент каталог → корзина → заказ

- Компонент переключателя + storage.
- Тело запроса при создании заказа.
- Превью сумм каталога/корзины через общую функцию из `lib`.

### Фаза 4 — редактирование и вторичные поверхности

- `greenwich-edit` / `warehouse-edit`.
- Карточка заявки, списки (если показывают период), сметы/проект (`ProjectEstimatePanel`, read-model seed).

### Фаза 5 — экспорт и коммуникации

- XLSX, Telegram, любые строки вида «N дней».

### Фаза 6 — Регрессия

- Создание Greenwich / внешняя заявка, проектная заявка, доп.-выдача.
- Один счётчик: до/после для каждого кейса матрицы.
- `npm run build`, при новых полях в API при необходимости `npm run brain:inventory` (route handlers могут не измениться, если только body).

## 10. Риски и как снимать

| Риск | Митигация |
|------|-----------|
| Неверная матрица дней → недобор/перебор денег | Фаза 0 + тесты на все утверждённые пары; code review только одного модуля формулы. |
| Расхождение клиент/сервер | Одна функция в `src/lib`, сервер импортирует тот же модуль. |
| Миграция исказила старые суммы при открытии заявки | Явный дефолт enum + заметка в релизе; при желании версионировать «правило расчёта дня» (избыточно на V1). |
| Резерв «на целые дни» vs оплата «полдней» | Согласовать с бизнесом; не смешивать без решения в ADR. |
| Дублирование стилей Uiverse по всему сайту | Один компонент + scoped классы. |

## 12. Итог

Качественная реализация **несёт умеренный риск** при условии: (1) закрытая матрица дней до кода; (2) одна точка расчёта; (3) данные сохранены на заказе. Делать **только в каталоге** без сервера **не рекомендуется** — ломает доверие к суммам.
