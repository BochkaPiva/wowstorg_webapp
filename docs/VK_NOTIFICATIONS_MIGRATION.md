# Переход уведомлений с Telegram на VK

Документ фиксирует полный план миграции уведомлений на VK с упором на:
- надёжную доставку в РФ;
- управление получателями через админку;
- отсутствие ручной работы с JSON для сотрудников.

## Цель

Перевести системные уведомления по заявкам на VK без риска:
- не потерять уведомления при переходе;
- не отправлять сообщения клиентам из рабочего сообщества;
- дать администратору полный контроль (кому, что и когда отправлять).

## Базовое решение

1. Использовать **отдельное VK-сообщество** для внутренних уведомлений (не клиентское).
2. Хранить привязку сотрудника к VK через `vkUserId` (аналог `chat_id` в Telegram).
3. Отправлять сообщения только по white-list сотрудников из админки.
4. На старте поддержать режим dual-run (VK основной / Telegram резерв или наоборот).

## Что нужно подготовить во VK

### 1) Сообщество
- Создать отдельное сообщество для бота уведомлений.
- Включить «Сообщения сообщества».

### 2) API-доступ
- Получить токен группы с правами `messages`.
- Включить Callback API.
- Указать адрес сервера: `https://<domain>/api/vk/callback` (после деплоя endpoint).

### 3) События Callback API (минимально необходимые)
- `message_new` — обязательно (привязка сотрудников и входящие команды).
- `message_allow` — рекомендуется (пользователь разрешил сообщения).
- `message_deny` — рекомендуется (пользователь запретил сообщения).

Остальные события на старте можно не включать.

## Что должно появиться в админке

Для полного контроля нужна отдельная зона управления VK-привязками:

1. Список сотрудников и статус VK:
   - привязан / не привязан;
   - уведомления включены / выключены;
   - дата последней активности.
2. Действия:
   - «Сгенерировать код привязки»;
   - «Отключить уведомления»;
   - «Отвязать VK»;
   - «Перепривязать».
3. Фильтры:
   - только непривязанные;
   - только отключённые;
   - по роли (`GREENWICH`, `WOWSTORG`, `ADMIN`).

## Как привязывать сотрудников без JSON

Рекомендуемый UX (как можно проще):

1. Админ в карточке сотрудника нажимает «Сгенерировать код».
2. Сотрудник открывает чат с сообществом и отправляет:
   - `Привязка 123456` (одноразовый код).
3. Система:
   - считывает `from_id` из события `message_new`;
   - связывает `from_id` с сотрудником;
   - подтверждает привязку ответным сообщением.

Плюсы:
- сотруднику не нужно искать ID вручную;
- нет работы с JSON и сторонними сервисами;
- меньше ошибок и быстрый onboarding.

## Правила безопасности отправки

Отправлять уведомления только если одновременно выполнены условия:

1. Пользователь есть в системе и имеет разрешённую роль.
2. У пользователя заполнен `vkUserId`.
3. У пользователя включён флаг `vkNotifyEnabled`.
4. Для события разрешён тип уведомления по роли (матрица доступа).

Иначе: не отправлять, писать причину в логи.

## Рекомендуемая модель данных

Минимум полей (в `User` или в отдельной таблице привязок):

- `vkUserId` (string/bigint, unique, nullable)
- `vkDisplayName` (nullable)
- `vkNotifyEnabled` (boolean, default true)
- `vkLinkedAt` (datetime, nullable)
- `vkLastIncomingAt` (datetime, nullable)

Для кодов привязки:
- `userId`
- `codeHash`
- `expiresAt`
- `usedAt`
- `createdBy`

## Переменные окружения (план)

- `VK_GROUP_ID`
- `VK_GROUP_TOKEN`
- `VK_CALLBACK_SECRET`
- `VK_CALLBACK_CONFIRMATION_TOKEN`
- `VK_NOTIFICATIONS_ENABLED=true|false`
- `VK_SEND_TIMEOUT_MS` (опционально)

При dual-run (переходный период):
- `NOTIFY_PRIMARY=vk|telegram`
- `NOTIFY_FALLBACK=telegram|none`

## Поэтапный план внедрения

### Этап 0. Подготовка (без деплоя не завершить)
- Подготовить сообщество и токен.
- Определить домен callback endpoint.
- Согласовать роли и матрицу уведомлений.

### Этап 1. Инфраструктура
- Добавить VK transport и callback endpoint.
- Реализовать проверку `secret` и `confirmation`.
- Включить логирование доставки и ошибок.

### Этап 2. Админка и привязки
- Добавить UI управления VK-привязками.
- Добавить генерацию одноразовых кодов.
- Добавить массовый экран статусов сотрудников.

### Этап 3. Dual-run
- Включить отправку в VK для тестовой группы сотрудников.
- Сохранить Telegram как резерв.
- Сравнить доставку и стабильность 3–7 дней.

### Этап 4. Переключение
- Сделать VK основным каналом.
- Telegram оставить резервом на время наблюдения.
- После стабилизации отключить Telegram-канал (по решению).

## Чеклист запуска в прод

Перед включением:
- [ ] Callback endpoint доступен по HTTPS.
- [ ] VK Callback подтверждён в кабинете VK.
- [ ] Секрет и confirmation token валидны.
- [ ] В админке есть статусы привязок.
- [ ] Есть минимум 2 тестовых сотрудника с успешной доставкой.
- [ ] Логи отправки/ошибок доступны и читаемы.

После включения:
- [ ] Проверить тестовые уведомления по ключевым событиям заявки.
- [ ] Проверить отключение уведомлений для одного сотрудника.
- [ ] Проверить отвязку/перепривязку.
- [ ] Проверить, что клиентские диалоги не затрагиваются.

## Откат (rollback)

Если в VK проблемы:

1. Переключить primary обратно на Telegram.
2. VK оставить выключенным флагом `VK_NOTIFICATIONS_ENABLED=false`.
3. Не удалять привязки — сохранить для повторного запуска.
4. Зафиксировать причины сбоев (таймаут, rate-limit, access denied, block).
