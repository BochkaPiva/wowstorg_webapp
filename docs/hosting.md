# Hosting (V1) — варианты и выбранная конфигурация

## Требования (из проекта)
- РФ-оплата/счёт: желательно без проблем.
- Минимум DevOps (managed DB, managed object storage, понятный деплой).
- Доступность из РФ: стабильная работа для сотрудников в РФ.
- Быстрые картинки: CDN + превью.
- Нагрузка маленькая (≈15 пользователей), но важна надёжность.
- Fallback: возможность быстро переключить источник картинок/статики (через env).

## Вариант A (рекомендуемый): Selectel Cloud (RU)
Почему:
- Хорошая практичность для РФ: оплата/документы, доступность.
- Есть managed Postgres, S3‑совместимое Object Storage, CDN.
- Можно держать Next.js как один Node‑сервис (SSR + API routes) на облачном сервере/контейнере.

Состав:
- **Next.js app**: 1 небольшой сервер (Node 20), деплой через Docker.
- **Postgres**: managed Postgres.
- **Object Storage (S3)**: оригиналы + превью.
- **CDN**: на бакет/префикс превью.
- **Наблюдаемость**: Sentry + структурные логи.
- **Бэкапы**:
  - Postgres: ежедневные managed snapshot + point-in-time (если доступно по тарифу).
  - Object storage: versioning/retention policy (если поддерживается) или периодический экспорт.

Риски/минусы:
- Деплой Next.js всё равно потребует минимального “серверного” контура (Docker/перезапуск).

## Вариант B: Yandex Cloud (RU)
Почему:
- Очень сильная экосистема managed сервисов.
- Managed Postgres + Object Storage + CDN, нормальная observability.

Состав:
- **Next.js app**: Compute VM (Docker) или Managed Containers (если удобно).
- Остальное аналогично варианту A.

Минусы:
- Чуть больше “инфраструктурных” вариантов и настроек (может занять больше времени на выбор оптимального).

## Вариант C (если хотим “как Vercel”): зарубежный PaaS + managed Postgres
Удобство максимальное, но:
- оплата/доступность/санкционные риски хуже
- домен/сертификаты не проблема, но платёжная и “долгоживущая” надёжность для РФ спорная

## Бюджет до 1000 ₽/мес (альтернатива Selectel)

Selectel только за VPS выходит ~1600 ₽/мес — выше целевого бюджета. Варианты уложиться в **до 1000 ₽/мес**:

### Стратегия: один VPS «всё в одном»
На одном недорогом VPS крутим:
- **Next.js** (Node 20, PM2 или Docker)
- **Postgres** (установлен на том же сервере, не managed)
- **Файлы (фото)** — локальный диск или папка (позже можно вынести в S3)

Плюсы: один платёж, обычно 300–800 ₽/мес. Минусы: бэкапы БД настраиваем сами (cron + дамп в файл/облако), нет отдельного managed Postgres.

### Провайдеры с низкими ценами (РФ, ориентировочно)
- **FirstVDS** — VPS от ~200–400 ₽/мес (1 vCPU, 1 ГБ RAM), 2 ГБ RAM — выше, но часто в пределах 500–700 ₽.
- **RuVDS** — недорогие тарифы, часто акции.
- **VDSina.ru** — низкий порог входа по цене.
- **Timeweb** — VPS/облако, бывают тарифы дешевле Selectel.
- **Reg.ru** — VPS, нередко есть промо/скидки на первый период.

Что проверить при выборе: наличие **публичного IP**, **доступ по SSH**, возможность поставить **Docker** или Node 20, оплата **из РФ** (карта/счёт).

### Решение V1 (текущее)
- **Бюджет до 1000 ₽/мес**: один VPS у FirstVDS / RuVDS / Timeweb / Reg.ru (1 vCPU, 2 ГБ RAM или 1 ГБ на старте), на нём Next.js + Postgres + файлы. Бэкапы БД — скриптом (ежедневный дамп).
- **Если бюджет вырастет**: перенос на Selectel/Yandex Cloud с отдельным managed Postgres и S3 (конфиг в env уже заложен).

## Ранее: Selectel (при бюджете 1500+ ₽/мес)
Выбираем **Вариант A (Selectel Cloud)** если решим поднять бюджет: отдельный VPS + managed Postgres + S3.

## Fallback-план по картинкам (обязательное требование)
Делаем так, чтобы в любой момент можно было переключить:
- `S3_ENDPOINT`
- `S3_BUCKET`
- `S3_REGION`
- `S3_PUBLIC_BASE_URL` (или `CDN_BASE_URL`)

Принцип:
- в БД храним **ключи объектов** (например `items/{itemId}/{uuid}.jpg`), а не абсолютные URL
- публичный URL собираем из env (CDN/endpoint), поэтому перенос бакета/провайдера не ломает данные

## Домен и HTTPS
Так как домена нет:
- покупаем домен у регистратора, который удобен по оплате из РФ
- DNS можно держать у регистратора или у облака
- TLS: автоматический (Let’s Encrypt) на входном reverse-proxy/ingress
