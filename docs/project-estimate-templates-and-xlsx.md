# Смета проекта: шаблоны разделов, колонки и два варианта XLSX

Документ фиксирует целевое поведение после доработки (январь 2026). Источник правды по коду — `prisma/schema.prisma`, `src/server/projects/estimate-read-model.ts`, `src/server/projects/estimate-xlsx.ts`, `src/app/projects/[id]/ProjectEstimatePanel.tsx`.

## 1. Цели

- Разделить **тип шаблона** (какие колонки доступны) и **название раздела в смете** (всегда задаётся вручную пользователем; в Excel/PDF попадает только `title`).
- **Реквизит** (`REQUISITE`): данные позиций по-прежнему приходят из связанной заявки; в смете добавляются **локальные** поля (например единица измерения), **не** синхронизируемые с `Order`.
- **Универсальный раздел** (в БД: `LOCAL`): те же клиентские колонки, что у реквизита в смете; строки хранятся в `ProjectEstimateLine`.
- **Подрядчики** (в БД: `CONTRACTOR`): все поля универсального раздела **плюс** внутренние поля (оплата, статус, комментарий, реквизиты подрядчика) — только для внутренней выгрузки и UI склада, не для клиента.
- **Два файла XLSX**: внутренний (полный) и клиентский (только нужные клиенту колонки + итоги по разделам, комиссия, итог с комиссией). PDF не развиваем.
- **Подытог по разделу** (выручка, внутр., налог 6%, маржа): только во **внутреннем** XLSX, отдельные строки под таблицей раздела; **не** дублировать этот пораздельный блок в веб-интерфейсе (глобальные итоги по проекту — внизу панели, см. ниже).
- **Итоги внизу панели сметы (веб):** сумма клиентских строк, условный налог 6% (от этой суммы), комиссия 15%, итого клиент, себестоимость, валовая маржа, маржа после налога и её % — формулы согласованы с глобальным хвостом **внутреннего** XLSX (`estimate-xlsx.ts` и `ProjectEstimatePanel.tsx`).
- Веб-таблица: **компактнее** (меньше padding/шрифт меток, узкие поля сумм там, где вводятся короткие числа), без изменения бизнес-логики.
- **Цвета в UI сметы:** блок **«Клиенту»** и карточка «Сумма (клиент)» — в фиолетово-индиго тонах (согласовано с брендом); блок **«Наши поля»**, раздел **«Подрядчики»** (фон секции и бейдж) — **нейтральный zinc/slate**, без сочетания голубого и жёлтого; кнопка «Сохранить смету» и полоса черновика — фиолетовый градиент; ссылка «XLSX клиент» — индиго.
- **Универсальный и подрядчики:** форма ввода **новой строки** по умолчанию **свёрнута**; раскрытие по кнопке «+ Добавить строку», после добавления строки форма снова сворачивается (меньше визуального шума при просмотре).
- **Реквизит, доп. услуги (доставка / монтаж / демонтаж):** комментарий и суммы **в одной строке** на широком экране (`sm+`); поля «Клиент ₽» и «Внутр. ₽» **узкие**, в том же визуальном ключе, что и поля позиций сметы (`cellXs`).

## 2. Типы разделов (enum)

| Значение Prisma   | Смысл для пользователя   | Связь с заявкой | Строки в БД (`ProjectEstimateLine`)        |
|-------------------|-------------------------|-----------------|--------------------------------------------|
| `REQUISITE`       | Реквизит                | `linkedOrderId` | Опционально; основной состав — из заказа   |
| `LOCAL`           | Универсальный           | нет             | да                                         |
| `CONTRACTOR`      | Подрядчики              | нет             | да                                         |

Переименование `LOCAL` → «универсальный» только в UI; значение enum не меняем (обратная совместимость с данными и миграциями).

## 3. Колонки строк (семантика)

### 3.1 Общие для реквизита (в смете), универсального и подрядчиков (клиент видит их в своём XLSX)

| Поле            | Описание |
|-----------------|----------|
| Позиция         | `name` (как сейчас) |
| Описание        | `description` |
| Ед. изм.        | локально для сметы; для реквизита — в `lineLocalExtras` по ключу id строки read-model |
| Кол-во          | для реквизита — из заявки (`qty`); для LOCAL/CONTRACTOR — `ProjectEstimateLine.qty` |
| Сумма за ед.    | клиентская цена за единицу (`unitPriceClient`); для реквизита **только чтение**, из расчёта заказа (см. §5) |
| Сумма по строке | `costClient` (как сейчас итог по строке для клиента) |

Инвариант для редактируемых строк (LOCAL/CONTRACTOR): при изменении `qty` или `unitPriceClient` UI пересчитывает `costClient` (округление до целых рублей); **ручной ввод суммы клиенту отключён** — только отображение `qty × unitPriceClient` (старые строки без пары qty/цена могут показывать сохранённый `costClient` до пересчёта).

### 3.2 Только внутренние (подрядчики + общий внутренний XLSX)

| Поле                    | Хранение |
|-------------------------|----------|
| Внутр. (себестоимость)  | `costInternal` (уже есть) |
| Оплата                  | `paymentMethod`: в UI — выбор «Наличные» / «Безнал» (select) |
| Статус оплаты           | `paymentStatus`: в UI — одно поле **input + datalist** (как комбобокс заказчика в корзине): можно выбрать «Оплачено» / «Не оплачено» или ввести свой текст; пресеты отображаются **цветом текста** (зелёный / красный), остальное — обычный цвет |
| Комментарий подрядчику  | `contractorNote` |
| Реквизиты (счёт и т.п.) | `contractorRequisites`, одна строка ввода (визуально компактно) |

Для разделов `LOCAL` и `REQUISITE` эти поля в UI и в БД не используются (NULL).

## 4. Локальные поля реквизита без строк в `ProjectEstimateLine`

Строки реквизита в read-model строятся из `Order` (аренда + услуги). Чтобы хранить **только сметочные** атрибуты (ед. изм.), используется JSON на разделе:

- `ProjectEstimateSection.lineLocalExtras`: объект `Record<string, { unit?: string | null }>`  
- Ключ — `id` строки из read-model (`orderLine.id` или `${orderId}:delivery` и т.д.).

Синхронизация с заявкой не затрагивает этот JSON. При удалении/замене заявки ключи могут устареть — допустимы «висячие» ключи; при отображении игнорируем отсутствующие id.

## 5. Реквизит: «цена за шт» и количество

- **Кол-во**: из заявки (`requestedQty` / логика услуг как сейчас).
- **Сумма за ед. (клиент)** для аренды:  
  `unitPriceClient = round(costClient / qty)` при `qty > 0`, где `costClient` — уже рассчитанная сумма строки (дни × цена за день × множитель), как в `estimate-read-model` сегодня.
- Поле **только для отображения** (и XLSX), не редактируется вручную; при изменении заявки пересчитывается автоматически.

Услуги (доставка и т.д.): ед. изм. по умолчанию можно считать «усл.» в UI, кол-во `1`, сумма за ед. = `costClient`.

## 6. Подытог раздела (только внутренний XLSX)

Для каждого раздела после строк раздела добавляется блок строк (не дублировать в UI):

Вычисления по строкам раздела (числа в валюте проекта, округление до целых рублей как в текущих итогах сметы):

- **Выручка (клиент), раздел** — сумма `costClient` по строкам.
- **Внутр., раздел** — сумма `costInternal` по строкам.
- **Валовая маржа** — выручка − внутр.
- **Условный налог 6%** — `round(выручка раздела × 0.06)` (налог с выручки раздела; при смене политики меняется только этот блок в коде/доке).
- **Маржа после условного налога** — валовая маржа − условный налог 6%.

Глобальные итоги в конце листа (внутренний вариант): агрегат по всем разделам + при необходимости дублирование комиссии 15% / итого клиент (как в текущей логике), плюс строки по **всему проекту**: суммарная выручка, суммарная внутр., валовая маржа, налог 6% от суммарной выручки, маржа после налога — согласованно с подытогами разделов.

Клиентский XLSX: для каждого раздела — только **итог по разделу** (сумма клиентских строк), затем общая сумма, комиссия 15%, итого с комиссией; без внутренних колонок и без налогов/маржи.

## 7. API (изменения)

- `POST /api/projects/[id]/estimate/sections`: тело `{ title, versionNumber?, kind?: 'LOCAL' | 'CONTRACTOR' }` (по умолчанию `LOCAL`). Название раздела всегда из `title`.
- `PATCH /api/projects/[id]/estimate/sections/[sectionId]`: опционально `lineLocalExtras` (merge/replace оговорить в коде — предпочтительно **полная замена** объекта с фронта при сохранении extras реквизита).
- `PATCH /api/projects/[id]/estimate`: расширить строки черновика полями `unit`, `qty`, `unitPriceClient`, contractor-поля для соответствующих разделов.
- `PATCH/POST` строк: расширить Zod и Prisma-поля.
- `GET /api/projects/[id]/estimate/pdf?version=&variant=client|internal` — `variant` по умолчанию `internal`; клиентский вариант отдаёт тот же content-type XLSX.

## 8. План внедрения (порядок работ)

1. **Prisma**: enum `CONTRACTOR`; поля строки; `lineLocalExtras` на секции; миграция.
2. **Read-model**: проброс `kind`, `lineLocalExtras`, новых полей строк; реквизит: `unit` из extras, `unitPriceClient` вычисляемо; `DRAFT_REQUISITE` без изменений логики заказа.
3. **XLSX**: `buildProjectEstimateXlsx({ variant, ... })` — два макета; внутренний — подытоги разделов и глобальные финансы.
4. **API**: секции, строки, черновик PATCH, копирование версий — перенос новых полей.
5. **UI**: выбор шаблона при добавлении раздела; компактная сетка; `LineEditor` / `AddLineForm` / реквизит; второй пункт скачивания «Для клиента».
6. **Регрессия**: сохранение сметы, импорт из заявок, demo-реквизит, итоги внизу панели (формулы не ломать).

## 9. Риски и ограничения

- Старые строки без `qty`/`unitPriceClient`: отображение как сейчас (только `costClient` / `costInternal`).
- «Висячие» ключи в `lineLocalExtras` после изменения состава заявки — допустимы.
- Налог 6% — **условная** модель для отчётности в XLSX, не налоговая консультация; при смене ставки правится константа в коде.
